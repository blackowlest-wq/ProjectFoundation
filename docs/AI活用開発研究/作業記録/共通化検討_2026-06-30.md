# 共通化検討 2026-06-30

## 結論

ログイン機能と日報機能で別々の DB 接続実装を持っているわけではない。
現状のバックエンドは Spring Boot の `spring.datasource.*` と JPA Repository を使っており、DB 接続機能自体はアプリ全体で共通である。

ただし、テスト実行時の接続情報、テスト補助処理、API クライアント、テストデータ投入、業務ルール実装には共通化不足がある。
今回「ログインでは Oracle、日報では H2 に見える」混乱が起きた主因は、DB 接続コードの分岐ではなく、テスト設定と接続情報の扱いが共通の実行入口として整理されていないことにある。

## 現状確認

- `backend/pom.xml` に H2 依存は存在しない。
- `backend/src/test/resources/application-test.yml` は Oracle JDBC を参照している。
- ログインテスト、日報テストとも `@ActiveProfiles("test")` を使い、同じ `application-test.yml` を読む。
- `spring.datasource.username` / `password` は `DAILY_REPORT_DB_USER` / `DAILY_REPORT_DB_PASSWORD` 環境変数に依存している。
- 環境変数がない、または誤っている場合は Oracle 接続で `ORA-01017` になり、H2 へフォールバックする構成ではない。

## 共通化が必要な箇所

| 優先度 | 対象 | 現状 | 推奨 |
| --- | --- | --- | --- |
| 高 | DB 接続設定の実行入口 | 環境変数を手動設定して `mvn` を実行する必要がある | `backend/scripts/test-oracle.ps1` に Oracle 接続設定と Maven 実行を集約 |
| 高 | テスト基底クラス | `@SpringBootTest`、`@AutoConfigureMockMvc`、`@ActiveProfiles("test")`、ログイン処理が各テストに分散 | `BaseMockMvcTest` または `MockMvcTestSupport` を作成 |
| 高 | MockMvc ログイン補助 | `AuthControllerTest` と `DailyReportControllerTest` に `loginAs` / JSON 作成が重複 | `TestLoginSupport` に集約 |
| 中 | CSRF 付き API 操作 | `.with(csrf()).session(session)` が多数重複 | `authenticatedPost` / `authenticatedPut` などのテストヘルパー化 |
| 中 | テストデータ初期化 | `DataInitializer` と `@Sql`、個別 `JdbcTemplate` 更新が混在 | テストデータ投入 SQL または Fixture クラスへ整理 |
| 中 | フロント API クライアント | `authApi.ts` と `dailyReportApi.ts` が直接 `fetch` を組み立てている | `apiClient.ts` に `getJson` / `postJson` / `putJson` / `postNoBody` を追加 |
| 中 | E2E API モック | `daily-report.spec.ts` に静的配信、認証、マスタ、日報 API モックが集中 | `e2e/support` 配下へ分割 |
| 中 | エラーコード | `AUTHENTICATION_FAILED`、`VALIDATION_ERROR` などが文字列直書き | バックエンドは enum/定数、フロントは union type で共有 |
| 低 | 業務ルールの型表現 | 休日区分、承認状態、勤務区分 ID が文字列中心 | enum/値オブジェクト化。ただし現時点では過剰になりすぎないよう注意 |

## 推奨する対応順

1. Oracle テスト実行スクリプトを使う。
   - 目的: 実行者ごとの差をなくし、H2 なのか Oracle なのかの混乱を防ぐ。
   - 実行入口: `backend/scripts/test-oracle.cmd`
   - PowerShell から直接使う場合: `backend/scripts/test-oracle.ps1`
   - ローカル設定: `backend/config/oracle-test.properties`
   - サンプル: `backend/config/oracle-test.example.properties`
   - 設定項目: `DAILY_REPORT_DB_URL`, `DAILY_REPORT_DB_USER`, `DAILY_REPORT_DB_PASSWORD`

2. MockMvc テスト共通基盤を追加する。
   - 目的: ログイン、CSRF、JSON 生成、セッション取得を全機能で同じ手順にする。
   - 候補:
     - `backend/src/test/java/com/example/dailyreport/support/BaseMockMvcTest.java`
     - `backend/src/test/java/com/example/dailyreport/support/TestJson.java`
     - `backend/src/test/java/com/example/dailyreport/support/TestUsers.java`

3. フロント API クライアントを拡張する。
   - 目的: `credentials: 'include'`、CSRF、JSON 読み取り、エラー処理を呼び出し側から隠す。
   - 候補:
     - `getJson<T>(url)`
     - `postJson<T>(url, body)`
     - `putJson<T>(url, body)`
     - `postJsonNoBody<T>(url)`

4. E2E support を分割する。
   - 目的: 日報以外の E2E を追加しても、ログインモックや静的配信を再利用できるようにする。
   - 候補:
     - `frontend/e2e/support/staticFrontend.ts`
     - `frontend/e2e/support/authMocks.ts`
     - `frontend/e2e/support/dailyReportMocks.ts`

## 注意点

- DB 接続設定を独自クラスで包みすぎる必要はない。Spring Boot の DataSource 自動設定を使う方針は維持する。
- 共通化の中心は「接続処理の再実装」ではなく、「設定・テスト実行・テスト補助・API 呼び出しの標準化」とする。
- Oracle 実機を使うテストと、将来もし軽量な単体テストを追加する場合は、実行プロファイルを明示的に分ける。
- Windows の PowerShell 実行ポリシーで `.ps1` が直接実行できない場合があるため、通常は `.cmd` ラッパーを使う。
- H2 を再導入する場合は Oracle との差異を受け入れる明確な目的が必要。現方針では Oracle 実機テストを正とする。
