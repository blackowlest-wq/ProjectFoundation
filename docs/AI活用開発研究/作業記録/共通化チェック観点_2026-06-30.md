# 共通化チェック観点 2026-06-30

## 目的

AI 駆動開発では、機能単位で実装を進めると、既存の共通部品を使わずに似た処理が再生成されやすい。
そのため、機能実装後に「共通化すべきか」と「作成済みコードが共通部品を使っているか」を確認する。

## チェック手順

1. 既存の共通部品を確認する。
   - バックエンド: 認証、例外応答、DB 設定、テスト補助、業務ルール
   - フロントエンド: API client、型定義、入力チェック、ルーティング、E2E support

2. 新規コードの直書きを探す。
   - `fetch`
   - `credentials: 'include'`
   - CSRF ヘッダー
   - MockMvc のログイン処理
   - `.with(csrf()).session(...)`
   - エラーコード文字列
   - テストデータ生成
   - E2E の `page.route`

3. 重複を分類する。
   - 完全に同じ責務: 共通化する
   - 似ているが業務意味が違う: 共通化しない
   - 今後増える見込みが高い: 小さな helper だけ作る
   - 仕様が揺れている: 記録して保留

4. 既存コードが共通部品を使っているか確認する。
   - 共通部品があるのに直接実装していないか
   - 呼び出し側に認証、CSRF、エラー形式の知識が漏れていないか
   - テストが機能ごとに別のログイン手順を持っていないか
   - E2E が同じモックや静的配信処理を各 spec に持っていないか

5. 共通化後に回帰確認する。
   - TypeScript 型チェック
   - フロントエンド単体テスト
   - E2E
   - バックエンド通常テスト
   - 必要に応じて coverage

## 今回のチェック結果

| 対象 | 判定 | 対応 |
| --- | --- | --- |
| フロント API 呼び出し | `authApi.ts` と `dailyReportApi.ts` に `fetch`、credentials、CSRF 指定が分散 | `apiClient.ts` に `getJson`、`postJson`、`postJsonWithCsrf`、`putJsonWithCsrf`、`postNoBodyWithCsrf` を追加 |
| E2E 静的配信 | `daily-report.spec.ts` に production build 配信処理が直書き | `frontend/e2e/support/staticFrontend.ts` へ分離 |
| E2E 認証モック | `daily-report.spec.ts` に認証 API モックとログイン操作が直書き | `frontend/e2e/support/authMocks.ts` へ分離 |
| E2E 日報モック | `daily-report.spec.ts` に日報 API モックが集中 | `frontend/e2e/support/dailyReportMocks.ts` へ分離 |
| ログイン画面 | `App.tsx` にログインフォーム、文言、入力属性、送信処理が直書き | `LoginForm.tsx` と `loginUi.ts` へ分離し、E2E ログイン操作も同じ文言定数を参照 |
| フロントのファイル配置 | `src` 直下にログイン、日報、共通部品が混在 | `src/auth`、`src/dailyReport`、`src/shared`、`src/app` に分割 |
| MockMvc ログイン補助 | `AuthControllerTest` と `DailyReportControllerTest` に重複 | 差分が大きいため今回は保留。次回 `support` パッケージへ移す |
| Oracle テスト実行入口 | 環境変数設定が手動 | 今回は保留。次回 `backend/scripts/test-oracle.ps1` を追加候補 |

## 今後のルール

- 新しいフロント API は、直接 `fetch` せず `apiClient.ts` の共通関数を使う。
- 新しいログイン画面関連の文言・入力属性は `loginUi.ts` / `loginValidation.ts` を使う。
- 新しい機能は `src` 直下に置かず、機能別フォルダまたは `shared` に配置する。
- 新しい E2E は、静的配信と認証モックを spec に直書きしない。
- 新しい MockMvc テストを追加する前に、ログイン補助と CSRF 補助の共通化を先に検討する。
- 共通化は「似ているから」ではなく、「同じ責務で、変更理由も同じ」場合に行う。
