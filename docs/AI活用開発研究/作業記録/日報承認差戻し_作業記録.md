## 採用した設計とケース方針

- 承認・差戻し対象は `PENDING` の日報だけとし、その他の状態はHTTP 409 / `INVALID_STATUS`で拒否する。
- 上長の担当グループ判定は既存の `manager_group_permissions` と `DailyReportAccessPolicy` を利用する。社員・管理者の承認・差戻しはUI操作を表示せず、APIは403とする。
- 承認・差戻しの状態更新と監査項目更新は同一トランザクションで扱い、更新直前に現在状態を再確認する。競合の原子性を保証する楽観ロックは対象外である。
- 差戻しコメントは前後空白を除いた値が1文字以上、最大1000文字とする。バックエンドが正規化と最終判定を行い、trim済み値を保存・返却する。UIは空文字・空白のみ・trim後1001文字を送信しない補助検証とし、直接APIの空白のみ・trim後1001文字はHTTP 400 / `VALIDATION_ERROR`かつDB不変、trim後1000文字は成功として確認する。
- 承認・差戻しの変更系APIは既存CookieセッションとCSRF契約を使う。CookieなしはHTTP 401 / `UNAUTHORIZED`、ログイン済みのCSRF未指定・不正はHTTP 403の既存Spring Security共通応答とし、いずれも状態・監査項目・コメントを変更しない。
- A-009の日報詳細レスポンスは状態だけでなく、承認・差戻しの監査情報を返すことを確認対象とする。
- 未承認一覧は担当グループかつ `PENDING` の日報だけを返し、0件は空一覧とする。
- ケースは `TC-APR-001`～`TC-APR-010`、実テストIDは `RT-APR-*`、将来の指摘IDは `FIND-APR-*` とし、名前空間を分離する。
