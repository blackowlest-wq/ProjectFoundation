# プロセス改善（Skill・レビューゲート強化）

## 目的

日報編集画面の実装をいったん破棄し、実装前確認、ケースレビュー、観点別専門レビュー、トレーサビリティ、標準化判定、最終品質ゲートの順序を再発防止可能な手順へ改める。

## 改善前のベースライン（RED）

- 正本テストケースと受入条件レビューが最新の画面仕様を反映しておらず、ケースレビューを実装開始の停止条件として扱えていなかった。
- 実装後にFullを先に実行し、その後に単一の汎用コードレビューを行ったため、指摘修正後の再実行が必要になった。
- 設計、テスト不足、期待結果、トレーサビリティ、セキュリティ等の観点を専用エージェントへ分離して実行せず、観点別の独立した判定記録がなかった。
- 指摘ごとのAC/TC/実テストID、標準化判定、反映先、保留理由、再確認条件が統一されていなかった。

## 対応

- 日報編集画面の未コミット実装、関連テスト、E2E、実装計画、作業記録の機能実装追記を破棄した。承認済み設計書のコミット `1c39072` は保持した。
- `projectfoundation-preflight-ja` に、正本ケース・受入条件レビュー、ID名前空間、トレーサビリティ表、観点別専門レビュー計画、ユーザー承認まで実装を開始しないHard Gateを追加した。
- `projectfoundation-review-ja` に、専門レビュー、統合レビュー、指摘ごとの標準化判定、レビュー後のFull再実行を必須化した。
- 上記Skillの正本（`docs/.../skills`）と実行配置（`.agents/skills`）を同一内容へ更新した。
- `開発フロー.md`、`実装前チェック表.md`、`実装後レビュー表.md`、`テストケースレビュー観点.md`、`AI専門レビュー用プロンプト定義.md`へ同じゲートと記録項目を反映した。
- 今回のプロセス指摘を `日報登録編集_指摘一覧.md` の `DR-PROC-*` として記録した。

## トレーサビリティ契約

| 対象 | ID名前空間 | 用途 |
| --- | --- | --- |
| 受入条件 | `AC-*` | 仕様上の受入条件 |
| 正本テストケース | `TC-*` | 実装前にレビューするケース |
| 実テスト | `RT-*` | `test` / `it` / E2E / Oracle等の実装 |
| 指摘 | `FIND-*` または機能別接頭辞 | レビュー・検証で検出した問題 |

## 再開時の停止条件

次回の日報編集画面実装では、以下がすべて揃い、ユーザー承認を得るまで本体コードを変更しない。

1. 受入条件、正本テストケース、過去指摘・障害の対応表を更新する。
2. 正本ケースレビューと受入条件レビューを実施し、未確認・対象外・保留を明記する。
3. 設計、テスト不足、期待結果、トレーサビリティの専門レビューを独立実行し、必要に応じてセキュリティ、可読性・配置、CI・品質ゲートも追加する。
4. ケースIDと実テストIDを先に割り当て、AC→TC→RT→指摘の対応を記録する。
5. 実装前の検証コマンド、レビュー後の最終ゲート、失敗時の扱いを計画へ記録する。

## 検証結果

- 改訂後Skillの正本・実行配置4ファイルを `quick_validate.py` で検証し、すべて `Skill is valid!` となった。
- `npm.cmd run lint:markdown` は73ファイル・0件、`git diff --check` も成功した。
- 改訂後Skillを新規コンテキストの読み取り専用作業シナリオへ適用した。正本ケースレビュー、AC/TC/RT/FINDの分離、観点別独立レビュー、統合レビュー、指摘修正後Full、標準化判定、記録更新、実装開始停止条件が計画に出力された。
- Forward-testではファイル変更がなく、改訂後Skillの実行結果に「既存の受入条件レビュー不足、正式な対応表不足、独立レビュー・統合レビュー記録不足、Oracle CI・依存監査の保留」が明示された。これは再開時に解消すべき停止条件として扱う。

## 2026-07-19 レビュー参照ファイルの固定化

### 追加で検出した問題

- Skillの手順に「テスト不足」「セキュリティ」などの観点名はあるが、セッションが変わったときに担当者が読む正本ファイル、必要な節、適用条件、出力先を必ず特定する契約がなかった。
- 現行Skillを新規コンテキストでForward-testした結果、担当者は一部の機能資料を列挙できたが、標準の`テストケースレビュー観点.md`や`AI専門レビュー用プロンプト定義.md`の担当観点別の節を必須入力として固定できなかった。
- 観点を全件実行するか、変更内容に応じて選択するかの優先度もSkill上で明示されていなかった。

### 対応

- 編集元である次のSkillへ、セッションをまたぐレビュー参照契約を追加した。
  - `docs/AI活用開発研究/構想メモ/標準化/skills/projectfoundation-preflight-ja/SKILL.md`
  - `docs/AI活用開発研究/構想メモ/標準化/skills/projectfoundation-review-ja/SKILL.md`
- 各Skillに、P0（常に適用）、P1（変更内容に応じて適用）、P2（明示要件またはリスクがある場合）の優先度を追加した。
- 設計、テスト不足、期待結果、トレーサビリティ、セキュリティ、配置・共通化、CI・品質ゲート、ログ・観測可能性について、具体的な標準ファイル、専門プロンプトの節、適用条件を表で固定した。
- `開発フロー.md`、`実装前チェック表.md`、`実装後レビュー表.md`にも、観点ごとの参照ファイル・節・P0/P1/P2の記録要件を反映した。
- 各レビュー依頼に、参照ファイル、参照節、対象範囲、出力先を明記し、観点名だけの依頼を未完了扱いにした。
- レビュー範囲、適用理由、各担当へ渡したファイル・節を、Planまたは専用レビュー記録の必須項目に追加した。
- 更新したSkillを正式配置へコピーした。
  - `.agents/skills/projectfoundation-preflight-ja/SKILL.md`
  - `.agents/skills/projectfoundation-review-ja/SKILL.md`

### 検証

- 編集元2ファイルと正式配置2ファイルを`quick_validate.py`で検証し、すべて`Skill is valid!`となった。
- 編集元と正式配置先のSHA-256一致を確認した。
- `npm.cmd run lint:markdown`は76ファイル・0件、`git diff --check`は終了コード0となった。
- 更新後Skillを新しいコンテキストでForward-testし、各観点の具体的な参照ファイル・節、P0/P1/P2、適用条件、記録先が出力されることを確認した。観点名だけの依頼にはならず、標準資料と専門プロンプト節が担当別計画へ展開された。
