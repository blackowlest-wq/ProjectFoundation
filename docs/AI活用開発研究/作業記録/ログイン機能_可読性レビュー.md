# ログイン機能 可読性レビュー

## 目的

リーダブルコードの原則に基づき、ログイン機能の実装、テスト、フロントエンドコードをレビューする。

## レビュー対象

- `backend/src/main/java/com/example/dailyreport/auth`
- `backend/src/main/java/com/example/dailyreport/config`
- `backend/src/main/java/com/example/dailyreport/common`
- `backend/src/test/java/com/example/dailyreport/auth`
- `frontend/src`
- `frontend/test`

## 基本チェックリスト

| 観点 | 判定 | メモ |
| --- | --- | --- |
| 変数・関数名は意図を明確に表現しているか | 一部改善余地あり | `logoutInvalidatesSession` がCSRF検証も含んでいる |
| 複数の解釈の余地がある命名を避けているか | 一部改善余地あり | テスト名と検証内容にズレがある |
| コードのグルーピングと整列は適切か | 一部改善余地あり | `App.tsx` と `SecurityConfig` に複数責務が集中している |
| コメントは「なぜ」を説明しているか | 該当少 | 複雑な背景コメントが必要な箇所は少ないが、CSRF除外理由は補足余地あり |
| 条件式は読みやすい順序になっているか | 一部改善余地あり | ロール別タイトルが三項演算子で横に長い |
| ネストは浅く、早期リターンを使っているか | 良好 | `DataInitializer` や `App.tsx` は早期returnを使えている |
| 変数のスコープは最小限か | 良好 | ローカル変数中心で不要な広域変数はない |
| 1つの関数/メソッドは1つの責務に集中しているか | 改善余地あり | `App`、`AuthController.login`、`SecurityConfig.securityFilterChain` |
| テストは読みやすく、意図が明確か | 一部改善余地あり | CSRF検証とログアウト検証を分けると読みやすい |

## カテゴリ別チェックリスト

### 命名規則

| 観点 | 判定 | メモ |
| --- | --- | --- |
| 変数・関数名は意図を明確に表現している | 一部改善余地あり | `App` は画面全体なので許容だが、内部関数が増えると曖昧になる |
| 汎用的すぎる名前を避けている | 良好 | `user`, `loading`, `loginId`, `password` は画面スコープでは十分 |
| 単位や属性を名前に含めている | 良好 | 現時点で時間・分などの値は少ない |
| スコープに応じて情報量を調整している | 良好 | 短いローカル変数中心 |
| 複数の解釈の余地がない命名になっている | 一部改善余地あり | `logoutInvalidatesSession` はCSRF検証を含むため曖昧 |
| 関数名は振る舞いや計算コストが想像できる | 良好 | `fetchMe`, `login`, `logout`, `readCookie` は分かりやすい |

### コードの構造

| 観点 | 判定 | メモ |
| --- | --- | --- |
| 関連するコードのグルーピングをしている | 一部改善余地あり | `App.tsx` にログイン画面とログイン後画面が同居 |
| 既存の似たコードと書き方を合わせている | 良好 | Spring Controller/Test、React hooks は一貫 |
| フォーマッターに通されて整列されている | 概ね良好 | 長いJSX行、長いテストJSONは改善余地あり |
| 処理の順序に合わせてコードを配置している | 良好 | 認証、セッション保存、レスポンス生成の順で並んでいる |

### コメント

| 観点 | 判定 | メモ |
| --- | --- | --- |
| コードを見れば分かる内容をコメントしていない | 良好 | 不要なコメントは少ない |
| 背景を説明している | 改善余地あり | `/api/auth/login` だけCSRF除外する理由は将来の保守者向けに補足余地あり |
| 変更可能性を説明している | 改善余地あり | テストユーザー投入条件やローカル用設定の境界を明文化するとよい |
| 代名詞を避け、主語を明確にしている | 該当少 | コメントが少ないため評価対象が少ない |
| 関数の外部振る舞いと内部詳細を分けている | 改善余地あり | 複雑化する前にメソッド分割で表現した方がよい |
| 複雑なロジックで具体例を書いている | 該当少 | 複雑なアルゴリズムはない |
| 情報密度の高い簡潔な表現を使っている | 良好 | 現状コメントは少なく冗長さはない |

### ロジック

| 観点 | 判定 | メモ |
| --- | --- | --- |
| 条件式は左辺に調査対象、右辺に基準値を書いている | 良好 | `response.status === 401` などは読みやすい |
| 優先度の高い条件から順に記述している | 良好 | `loading`、未ログイン、ログイン済みの順 |
| 早期リターンを使っている | 良好 | `DataInitializer`、`App` で利用 |
| ネストを浅くしている | 良好 | 深いネストは少ない |
| 巨大なコードブロックを分割している | 改善余地あり | `App.tsx` はコンポーネント分割余地あり |
| 不要な中間変数を作っていない | 良好 | おおむね簡潔 |
| 説明変数を適切に使っている | 一部改善余地あり | ロール別タイトルやテストJSONは説明変数化すると読みやすい |

### 変数

| 観点 | 判定 | メモ |
| --- | --- | --- |
| 変数のスコープを最小限にしている | 良好 | 関数内ローカル変数中心 |
| グローバル変数を避けている | 良好 | モジュール定数は限定的 |
| 変数はできるだけイミュータブルにしている | 一部改善余地あり | `readJson` の `let error` は早期returnまたは補助関数で不変化できる |

## 指摘一覧

### R-LOGIN-001 Appコンポーネントに複数責務が集まっている

- 重要度: Medium
- 対象: `frontend/src/App.tsx`
- 該当箇所: ログイン状態復元、ログイン処理、ログアウト処理、ログイン画面、ログイン後画面、ロール別タイトル表示
- 問題: 1ファイル・1コンポーネントに複数の関心事が集まっており、画面追加時に読み手が状態管理と表示構造を同時に追う必要がある。
- 推奨: `LoginForm`、`AuthenticatedHome`、`roleTitle` のように分割する。

### R-LOGIN-002 ロール別タイトルの三項演算子が横に長い

- 重要度: Low
- 対象: `frontend/src/App.tsx`
- 該当箇所: ロール別の `h1`
- 問題: 三項演算子が横に長く、ロール追加時の変更箇所が読み取りにくい。
- 推奨: `const pageTitleByRole: Record<Role, string>` のような対応表にする。

### R-LOGIN-003 AuthController.loginの責務がやや多い

- 重要度: Medium
- 対象: `backend/src/main/java/com/example/dailyreport/auth/AuthController.java`
- 該当箇所: `login`
- 問題: 認証、SecurityContext生成、セッションID再発行、セッション保存、レスポンス変換が1メソッドに並んでいる。
- 推奨: 認証後セッション確立処理を `establishSession` などのprivateメソッドへ切り出す。

### R-LOGIN-004 SecurityConfigのフィルタ設定が読み取りにくい

- 重要度: Low
- 対象: `backend/src/main/java/com/example/dailyreport/config/SecurityConfig.java`
- 該当箇所: `securityFilterChain`
- 問題: CSRF、認可、ログイン無効化、例外レスポンス生成が1つの長いチェーンに含まれている。
- 推奨: `writeForbiddenResponse` や `accessDeniedHandler` をprivateメソッド/Bean化する。

### R-LOGIN-005 DataInitializerのAppUser生成が位置引数に依存している

- 重要度: Medium
- 対象: `backend/src/main/java/com/example/dailyreport/config/DataInitializer.java`
- 該当箇所: 初期ユーザー3件の生成
- 問題: `AppUser` のコンストラクタ引数が多く、`null` が何の項目なのか読み手が追いにくい。
- 推奨: テストデータ用のファクトリメソッド、Builder、または名前付きの説明変数を使う。

### R-LOGIN-006 バリデーションエラーの詳細メッセージが実際の制約を表していない

- 重要度: Medium
- 対象: `backend/src/main/java/com/example/dailyreport/common/ApiExceptionHandler.java`
- 該当箇所: `validation`
- 問題: `@NotBlank` と `@Size` があるが、どのエラーでも詳細が「必須項目です。」になる。読み手にも利用者にも制約意図が伝わりにくい。
- 推奨: `error.getDefaultMessage()` を使うか、制約ごとのメッセージを定義する。

### R-LOGIN-007 テスト名と検証内容が一致しきっていない

- 重要度: Low
- 対象: `backend/src/test/java/com/example/dailyreport/auth/AuthControllerTest.java`
- 該当箇所: `logoutInvalidatesSession`
- 問題: メソッド名はセッション無効化を示すが、CSRFなし403とCSRFあり204も検証している。
- 推奨: `logoutRequiresCsrfToken` と `logoutWithCsrfInvalidatesSession` に分ける。

### R-LOGIN-008 テストリクエストJSONの文字列連結が読みづらい

- 重要度: Low
- 対象: `backend/src/test/java/com/example/dailyreport/auth/AuthControllerTest.java`
- 該当箇所: 長すぎるログインIDのテスト
- 問題: JSON文字列連結により、リクエストの意図より文字列構築の記述が目立つ。
- 推奨: `ObjectMapper`、ヘルパーメソッド、または小さな `loginJson(loginId, password)` を使う。

### R-LOGIN-009 readJsonのエラー組み立てが可変変数に依存している

- 重要度: Low
- 対象: `frontend/src/authApi.ts`
- 該当箇所: `readJson`
- 問題: `let error` を後から差し替える流れより、分岐ごとに返す/投げる形の方が読みやすい。
- 推奨: `readError(response)` を切り出し、`throw await readError(response)` のようにする。

## 総評

現状のログイン機能は、命名やネストの深さはおおむね良好で、致命的に読みにくい構造ではない。
一方で、初期実装として1ファイル・1メソッドに寄せた箇所が残っているため、次機能を追加する前に `App.tsx`、`AuthController.login`、`DataInitializer`、`AuthControllerTest` を小さく分割すると、今後の変更が追いやすくなる。

## 対応結果

| 指摘ID | 対応 |
| --- | --- |
| R-LOGIN-001 | `LoginForm` と `AuthenticatedHome` に分割 |
| R-LOGIN-002 | `pageTitleByRole` の対応表へ変更 |
| R-LOGIN-003 | `establishAuthenticatedSession` へ切り出し |
| R-LOGIN-004 | `accessDeniedHandler` と `writeForbiddenResponse` へ切り出し |
| R-LOGIN-005 | `employee`、`manager`、`admin` の初期データ生成メソッドへ分割 |
| R-LOGIN-006 | 制約別メッセージを定義し、`error.getDefaultMessage()` を返すよう変更 |
| R-LOGIN-007 | `logoutRequiresCsrfToken` と `logoutWithCsrfInvalidatesSession` に分割 |
| R-LOGIN-008 | `loginJson` ヘルパーでJSON生成を集約 |
| R-LOGIN-009 | `readError` を切り出し、`readJson` の可変変数を削除 |
