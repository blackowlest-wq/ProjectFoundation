# 日報承認・差戻し 正本テストケース

## ケース契約

- 機能: 日報承認・差戻し、未承認一覧、日報詳細
- ケースID: `TC-APR-001`～`TC-APR-010`
- 実テストID: `RT-APR-*`。実装後にテスト名またはテストコメントへ対応付ける
- 指摘ID: `FIND-APR-*`。ケースID・実テストIDとは再利用しない
- 状態表記: `DRAFT`=未提出、`PENDING`=承認待ち、`REJECTED`=差戻し、`APPROVED`=承認済み
- 期待結果は画面、HTTPレスポンス、DB列、API呼び出し有無などの観測可能な値だけを記載する
- 「対象外・保留」はケースごとに仕様外の期待値を追加しないための欄である

## 詳細ケース

### TC-APR-001 承認対象の上長がPENDING日報を承認できる

- 受入条件: F-007-AC-001～004、F-007-AC-009
- Given: 上長がログインし、担当グループの日報が `PENDING` で存在する
- When: `POST /api/daily-reports/{reportId}/approve` を既存CookieセッションとCSRFヘッダー付きで実行する
- Then: HTTP 200。レスポンスの `reportId` は対象ID、`approvalStatus` は `APPROVED`、`approverId`/`approverName`/`approvedAt` が存在する。DBの状態と3監査列が更新され、同一日報の詳細APIにも状態・監査情報が返る。担当上長以外のデータは変更されない
- テストデータ: 担当グループ、`PENDING`、操作者名スナップショットが確認できる日報
- 実テスト: RT-APR-BE-001 / `DailyReportApprovalControllerTest#rtAprBe001ManagerApprovesPendingReportAndDetailReturnsApprovalAudit`
- 対象外・保留: 日時の丸め・表示精度は既存日時契約。承認履歴の追加は対象外

### TC-APR-002 承認の状態・権限エラーは拒否されDB不変

- 受入条件: F-007-AC-005～008
- Given: 上長に `DRAFT`、`REJECTED`、`APPROVED`、担当外 `PENDING` の日報があり、社員・管理者もログインできる。別にCookieセッションなしの呼出し、ログイン済みでCSRFヘッダー未指定・不正の呼出しを用意する
- When: 各利用者または各セッション条件で承認APIを実行する
- Then: CookieなしはHTTP 401 / `UNAUTHORIZED`。ログイン済みのCSRF未指定・不正はHTTP 403かつ既存Spring Security共通応答。`DRAFT`/`REJECTED`/`APPROVED` はHTTP 409かつ `INVALID_STATUS`、担当外・社員・管理者はHTTP 403。各失敗前後で状態、承認監査項目、差戻し監査項目、コメントが不変である。権限外日報の内容・監査情報はレスポンスに含まれない
- 実テスト: RT-APR-BE-002 / `DailyReportApprovalControllerTest#rtAprBe002ApproveRejectsAllNonPendingAndOutsideReportsWithoutChanges`、RT-APR-BE-009 / `DailyReportApprovalControllerTest#rtAprBe009ApproveRequiresAuthenticationMissingAndInvalidCsrfAndRolesWithoutChanges`
- 対象外・保留: 競合更新の原子性は対象外。共通エラー本文の追加フィールドは既存契約

### TC-APR-003 未承認一覧から承認し画面が更新される

- 受入条件: F-007-AC-009～010
- Given: 上長が未承認一覧を開き、担当グループの `PENDING` 日報が表示される
- When: 対象日報の詳細へ遷移し、承認確認を完了して承認する
- Then: 詳細画面の表示状態が `APPROVED` に更新され、承認者と承認日時が表示され、`PENDING` 用の承認・差戻し操作は表示されない。一覧へ戻ると対象は未承認一覧の表示対象から外れる
- 実テスト: RT-APR-E2E-001 / `frontend/e2e/approval-rejection.spec.ts`
- 対象外・保留: 文言・レイアウトの完全一致は設計にない

### TC-APR-004 コメント付き差戻しが監査項目を保存する

- 受入条件: F-008-AC-001～005、F-008-AC-011
- Given: 担当グループの `PENDING` 日報に上長がログインしている
- When: `POST /api/daily-reports/{reportId}/reject` に `Content-Type: application/json`、既存Cookieセッション、CSRFヘッダーと `{"rejectComment":"  作業時間を確認してください。  "}` を送る
- Then: HTTP 200。`approvalStatus` は `REJECTED`、`rejectorId`/`rejectorName`/`rejectedAt`/`rejectComment` が返り、`rejectComment` は `作業時間を確認してください。` である。DBの状態、差戻し3監査列、最新コメントには同じtrim済み値が保存され、詳細APIにも同じtrim済み最新コメントと監査情報が返る
- 実テスト: RT-APR-BE-003 / `DailyReportApprovalControllerTest#rtAprBe003ManagerRejectsPendingReportAndDetailReturnsRejectionAudit`
- 対象外・保留: コメント履歴は追加しない。日時の厳密な形式は既存契約

### TC-APR-005 差戻しの状態・権限エラーは拒否されDB不変

- 受入条件: F-008-AC-006、F-008-AC-008～010
- Given: `DRAFT`、`REJECTED`、`APPROVED`、担当外 `PENDING` の日報、および社員・管理者のセッションがある。別にCookieセッションなしの呼出し、ログイン済みでCSRFヘッダー未指定・不正の呼出しを用意する
- When: 各利用者または各セッション条件で差戻しAPIを実行する
- Then: CookieなしはHTTP 401 / `UNAUTHORIZED`。ログイン済みのCSRF未指定・不正はHTTP 403かつ既存Spring Security共通応答。不正状態はHTTP 409 / `INVALID_STATUS`、担当外・社員・管理者はHTTP 403。各失敗前後で状態、差戻し監査項目、コメント、承認監査項目は不変である
- 実テスト: RT-APR-BE-004 / `DailyReportApprovalControllerTest#rtAprBe004RejectRejectsAllNonPendingAndOutsideReportsWithoutChanges`、RT-APR-BE-010 / `DailyReportApprovalControllerTest#rtAprBe010RejectRequiresAuthenticationMissingAndInvalidCsrfAndRolesWithoutChanges`
- 対象外・保留: 既存共通エラーの詳細本文は実装時に確認

### TC-APR-006 差戻しコメントのUI検証とバックエンド最終検証を確認する

- 受入条件: F-008-AC-002、F-008-AC-007
- Given: 上長が担当グループの `PENDING` 日報を開いている。直接API検証では有効なCookieセッションとCSRFヘッダーを付け、各入力前のDB状態・監査項目・コメントを記録する
- When (UI): 未入力、空文字、空白のみ、trim後1001文字の入力で確定する
- Then (UI): 画面で入力エラーを表示し、差戻しAPIを呼ばず、DBを変更しない
- When (直接API): 空白のみ、または前後空白を含めてもtrim後1001文字となるコメントを `POST /reject` へ送る。別にtrim後ちょうど1000文字のコメントを送る
- Then (直接API): 空白のみ・trim後1001文字はHTTP 400 / `VALIDATION_ERROR`で、状態、承認・差戻し監査項目、コメントは不変。trim後1000文字はHTTP 200で差戻しに成功し、保存・返却・詳細APIのコメントはtrim済み1000文字である。UI検証は補助であり、バックエンドが正規化と最終判定を行う
- 実テスト: RT-APR-BE-005 / `DailyReportApprovalControllerTest#rtAprBe005RejectCommentBoundaryValidationLeavesInvalidRowsUnchangedAndStoresTrimmedMaximum`、RT-APR-UI-002 / `DailyReportApprovalPanel.test.tsx#showsRejectCommentValidationAndDoesNotCallApi`
- 対象外・保留: 具体的なエラー文言の完全一致は要求しない

### TC-APR-007 差戻しから社員の修正・再提出へ接続する

- 受入条件: F-008-AC-011～012、F-009-AC-001～005
- Given: 上長が一覧から `PENDING` 日報を差し戻し、社員がその日報の所有者としてログインできる
- When: 社員が詳細で最新コメントを確認し、既存編集画面で修正して再提出する
- Then: 差戻し後は `REJECTED` とコメントが表示され、編集・再提出へ遷移できる。再提出後は `PENDING`、提出日時が更新され、最新差戻しコメントを参照でき、社員は承認待ち日報を編集できない
- 実テスト: RT-APR-E2E-002 / `frontend/e2e/approval-rejection.spec.ts`
- 対象外・保留: 再提出の入力チェック詳細はF-009既存ケースを再利用

### TC-APR-008 未承認一覧は担当グループのPENDINGだけを返す

- 受入条件: F-010-AC-001～004、F-010-AC-007～008
- Given: 上長の担当・担当外、`PENDING`・他状態、指定日付・社員・グループの組合せが存在する
- When: `GET /api/daily-reports/pending-approvals` を日付範囲、グループID、社員IDで検索し、画面で検索する
- Then: HTTP成功時の各一覧項目は担当グループかつ `PENDING` で、条件に一致しない項目は含まれない。複数件は`reportDate`、`employeeId`昇順で返る。担当外group指定と許可グループ0件は空一覧で、上長の日付未指定・逆順・366日超過はHTTP 400 / `VALIDATION_ERROR`である。担当外データは内容・監査情報を返さない
- 実テスト: RT-APR-BE-006 / `DailyReportApprovalControllerTest#rtAprBe006PendingApprovalsFilterPermittedGroupAndEmployeeOrderOnlyPendingReports`、`#rtAprBe006PendingApprovalsReturnEmptyArrayForManagerWithoutPermittedGroups`、`#rtAprBe006PendingApprovalsRejectMissingReverseAndOver366DayManagerRanges`、RT-APR-UI-001 / `DailyReportApprovalPanel.test.tsx`
- 対象外・保留: 日付範囲の上限・必須ルールは既存一覧API契約を使用

### TC-APR-009 社員・管理者は承認系入口を利用できない

- 受入条件: F-010-AC-005～006、F-007-AC-007～008、F-008-AC-009～010
- Given: 社員または管理者がログインしている
- When: 未承認一覧画面を開き、直接承認・差戻しAPIを試みる
- Then: 未認証かつ日付未指定のAPIはHTTP 401 / `UNAUTHORIZED`で入力検証より認証を優先する。UIには社員・管理者の未承認一覧の承認操作入口を表示せず、両ロールのAPIは日付指定有無にかかわらずHTTP 403。対象日報の状態・監査項目・コメントは不変である
- 実テスト: RT-APR-BE-007 / `DailyReportApprovalControllerTest#rtAprBe007PendingApprovalsAuthenticateBeforeDateValidationAndRejectNonManagers`、RT-APR-UI-003 / `DailyReportApprovalPanel.test.tsx`
- 対象外・保留: 管理者の日報参照はF-005の受入条件に従う

### TC-APR-010 詳細API・画面は認可範囲と監査情報を表示する

- 受入条件: F-005-AC-001～009
- Given: 社員本人、担当グループの上長、管理者、担当外の上長、存在しないIDの各条件がある。承認済み・差戻し済み日報には該当する監査情報がある
- When: `GET /api/daily-reports/{reportId}` を呼び、詳細画面を開く
- Then: 実際に承認または差戻し済みの日報について、社員本人、担当上長、管理者は認可範囲で日報ヘッダ、全作業明細、状態、提出日時、該当する承認者・承認日時または差戻し者・差戻し日時・最新コメントを確認できる。担当外上長は403で監査項目を返さず、存在しないIDは404である。上長かつ `PENDING` のときだけ操作を表示する
- 実テスト: RT-APR-BE-008 / `DailyReportApprovalControllerTest#rtAprBe008ApprovedDetailAllowsOwnerAssignedManagerAndAdminWithoutLeakingToOutsideManager`、`#rtAprBe008RejectedDetailAllowsOwnerAssignedManagerAndAdminWithoutLeakingToOutsideManager`、RT-APR-UI-004 / `DailyReportApprovalPanel.test.tsx`
- 対象外・保留: nullのJSON表現、日時の厳密形式、画面文言は設計・既存共通契約で確定
