# 日報登録・編集画面 セキュリティレビュー

## 対象

- 日報登録、取得、編集、提出、再提出 API
- 日報登録・編集画面
- フロントエンド API クライアント

## レビュー結果

| 観点 | 判定 | 内容 |
| --- | --- | --- |
| 認証 | 対応 | 既存 SecurityConfig により認証 API 以外は認証必須 |
| 認可 | 対応 | 登録・編集・提出・再提出は EMPLOYEE のみ。編集・提出・再提出は社員本人の日報のみ |
| CSRF | 対応 | 更新系 API は CSRF 保護対象。フロントは `XSRF-TOKEN` を `X-XSRF-TOKEN` として送信 |
| 状態遷移 | 対応 | DRAFT のみ提出、REJECTED のみ再提出、DRAFT/REJECTED のみ編集 |
| 入力検証 | 対応 | バックエンドを正として休日区分、時刻、作業時間、合計一致を検証 |
| 保存済みデータ整合性 | 修正済み | 提出・再提出時に保存済み日報を再検証する処理を追加 |
| 機密情報露出 | 対応 | 日報 API、フロント API でパスワード・ハッシュを扱わない |
| XSS | 概ね対応 | React の通常描画のみで、`dangerouslySetInnerHTML` は未使用 |
| ID 推測 | 一部保留 | UUID ベースの ID だが、GET で社員以外のロールに対する詳細認可は今後の一覧・承認機能で追加設計が必要 |
| エラー情報 | 概ね対応 | 共通エラー形式を使用。内部例外やスタックトレースは返さない |

## 修正した問題

| ID | 問題 | 修正 |
| --- | --- | --- |
| SEC-DR-001 | 提出・再提出時に保存済み日報の入力整合性を再検証していなかった | `TimeRules.validateStoredReport` を追加し、`submit` と `resubmit` で実行 |
| SEC-DR-002 | API クライアントの CSRF/JSON/エラー処理が重複し、変更漏れリスクがあった | `apiClient.ts` に共通化 |
| SEC-DR-003 | CSRF なし登録 API のテストが不足 | `createRequiresCsrfToken` を追加 |

## 保留事項

- 上長・管理者の詳細参照認可は、承認対象グループと全件参照の実装時に再レビューする
- Cookie の Secure / SameSite 属性は既存 SecurityConfig と環境設定に依存するため、本番相当環境で確認する
- Oracle 実機で DB 制約、DDL、権限設定を確認する
