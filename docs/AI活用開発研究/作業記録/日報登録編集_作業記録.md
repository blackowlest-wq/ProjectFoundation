# 日報登録・編集画面 作業記録

## 対象

- 日報登録・編集画面
- 日報登録、詳細取得、編集、提出、再提出 API
- 登録・編集画面で必要な案件、作業分類、休日区分マスタ API
- Oracle 用 DDL 差分

## 工数

| 区分 | 工数 |
| --- | ---: |
| 実装工数 | 4.8h |
| テスト作成工数 | 1.6h |
| 環境構築・DB・テストデータ準備工数 | 0.7h |

## 実装内容

- Spring Boot に `report` パッケージを追加
- `POST /api/daily-reports`、`GET /api/daily-reports/{reportId}`、`PUT /api/daily-reports/{reportId}`、`POST /api/daily-reports/{reportId}/submit`、`POST /api/daily-reports/{reportId}/resubmit` を追加
- `GET /api/master/projects`、`GET /api/master/work-categories`、`GET /api/master/holiday-types` を追加
- 社員のみ登録・編集・提出・再提出できるようにバックエンドで権限制御
- Cookie セッション認証と CSRF 前提で保護 API として実装
- 同一社員・同一日の日報重複を DB 制約とアプリケーションで防止
- 休日区分別の勤務時刻・作業明細ルール、時刻順、休憩自動計算、作業明細合計一致を実装
- React に日報登録・編集フォーム、作業明細追加・削除、保存、保存して提出を追加
- Oracle 用 `daily_reports`、`daily_report_work_items` DDL を追加

## 実装時の仮定

- 今回の主対象は日報登録・編集画面のため、承認、差戻し、一覧、月次集計、CSV 出力は実装対象外とした
- 既存ログイン実装がマスタを `users` に一部保持しているため、今回の案件、作業分類、休日区分は初期サンプルの固定マスタとして実装した
- 休憩区分は `BT001` を 12:00-13:00、`BT002` を 12:00-13:00 と 17:30-17:45 として扱った
- 勤務区分は `WT001` を 09:00-18:00、`WT002` を 09:00-17:30 として扱った
- 深夜時間帯は設計書どおり 22:00-05:00 とした
- 提出、再提出時の入力再検証は、登録・編集時に妥当な状態だけ保存される前提で、保存済み値の状態遷移検証を中心にした
