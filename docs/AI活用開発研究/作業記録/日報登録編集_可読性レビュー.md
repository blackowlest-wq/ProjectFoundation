# 日報登録・編集画面 可読性レビュー

## 目的

ログイン機能の可読性レビュー観点を使い、日報登録・編集画面のバックエンド、フロントエンド、テストコードをレビューする。

## レビュー対象

- `backend/src/main/java/com/example/dailyreport/report`
- `backend/src/test/java/com/example/dailyreport/report`
- `frontend/src/App.tsx`
- `frontend/src/dailyReportApi.ts`
- `frontend/src/dailyReportValidation.ts`
- `frontend/test/dailyReportApi.test.ts`
- `frontend/test/dailyReportValidation.test.ts`

## 基本チェックリスト

| 観点 | 判定 | メモ |
| --- | --- | --- |
| 変数・関数名が意味を明確に表現している | 一部改善余地あり | `save(thenSubmit)` は呼び出し側の意図が読み取りにくい |
| 複数の解釈や広すぎる名前を避けている | 一部改善余地あり | `TimeRules` が検証、計算、マスタ分岐を持つ |
| コードのグルーピングと整列が適切 | 一部改善余地あり | `DailyReportForm` に状態管理、API、描画が集中 |
| コメントは「なぜ」を説明している | 該当少 | 複雑な時間計算の補足コメントまたは名前付き構造が不足 |
| 条件式が読みやすい | 一部改善余地あり | 休日区分、時刻、作業明細の条件が `calculate` に集中 |
| ネストが浅く早期リターンを使っている | 概ね良好 | 早期 return は使われている |
| 1つの関数/メソッドが1つの責務に集中している | 改善余地あり | `DailyReportForm`、`TimeRules.calculate`、統合テストに責務集中 |
| テストが読みやすく意図が明確 | 一部改善余地あり | 正常フロー確認が1テストにまとまりすぎている |
| 既存コードのスタイルと一貫性がある | 概ね良好 | Controller/Service/Repository、Vitest 構成は既存方針に沿っている |

## 指摘一覧

### R-DR-001 `DailyReportForm` に複数責務が集中している

- 重要度: Medium
- 対象: `frontend/src/App.tsx`
- 該当箇所: `DailyReportForm`
- 問題: マスタ読込、既存日報読込、URL 解析、入力状態、休日区分制御、保存・提出 API、作業明細 UI、メッセージ表示が1コンポーネントにまとまっている。今後、差戻しコメント、詳細画面、一覧導線を追加すると読み手が状態管理と描画構造を同時に追う必要がある。
- 推奨: `DailyReportForm`、`WorkItemsEditor`、`useDailyReportEditor` などに分け、API/状態管理と JSX を分離する。

### R-DR-002 `TimeRules.calculate` が検証と計算を同時に担っている

- 重要度: Medium
- 対象: `backend/src/main/java/com/example/dailyreport/report/TimeRules.java`
- 該当箇所: `calculate`
- 問題: 時刻パース、休日区分別の入力可否、社員設定チェック、休憩計算、実勤務計算、作業明細合計チェック、勤務時間内訳計算が1メソッド内に並んでいる。設計書の業務ルール対応関係を追いにくい。
- 推奨: `validateHolidayTypeRule`、`validateWorkTimeRule`、`calculateBreakMinutes`、`splitWorkMinutes` のように業務ルール単位へ分割する。

### R-DR-003 時刻・時間帯の魔法値と `int[]` が意味を隠している

- 重要度: Medium
- 対象: `backend/src/main/java/com/example/dailyreport/report/TimeRules.java`
- 該当箇所: `720`、`780`、`1050`、`1065`、`1320`、`300`、`List<int[]>`
- 問題: 12:00、13:00、17:30、17:45、22:00、05:00 を分で直接表しており、読み手が都度変換する必要がある。`int[]` も開始・終了のどちらかを型から判断できない。
- 推奨: `TimePeriod` record と `minutes("12:00")` または名前付き定数を使う。

### R-DR-004 登録・編集時の検証と適用で同じ計算を二度呼んでいる

- 重要度: Low
- 対象: `backend/src/main/java/com/example/dailyreport/report/DailyReportService.java`
- 該当箇所: `validateRequest`、`apply`
- 問題: `validateRequest` で `TimeRules.calculate` を呼び、直後の `apply` でも再度 `TimeRules.calculate` を呼んでいる。読み手には「検証結果と適用結果が常に同一か」「副作用はないか」を余分に確認させる。
- 推奨: `validateAndCalculate` のように計算結果を返し、登録・編集処理でその結果を `applyContent` に渡す。

### R-DR-005 API 共通処理が `authApi.ts` と `dailyReportApi.ts` に重複している

- 重要度: Low
- 対象: `frontend/src/authApi.ts`、`frontend/src/dailyReportApi.ts`
- 該当箇所: `readCookie`、`readError`、`readJson`
- 問題: Cookie 読取、エラー変換、JSON 読取が2ファイルに存在する。今後、401/403 の文言や CSRF ヘッダーの扱いを変更すると差分漏れが起きやすい。
- 推奨: `apiClient.ts` などに共通化し、認証 API と日報 API はエンドポイント固有処理に集中させる。

### R-DR-006 日報 Controller にマスタ API と日報 API が同居している

- 重要度: Low
- 対象: `backend/src/main/java/com/example/dailyreport/report/DailyReportController.java`
- 該当箇所: `projects`、`workCategories`、`holidayTypes`、日報 CRUD/状態変更
- 問題: 初期実装としては小さいが、マスタ API と日報 API は関心が異なる。API が増えると Controller の見通しが落ちる。
- 推奨: `MasterController` と `DailyReportController` に分ける。

### R-DR-007 正常系テストが1ケースにまとまりすぎている

- 重要度: Low
- 対象: `backend/src/test/java/com/example/dailyreport/report/DailyReportControllerTest.java`
- 該当箇所: `createEditGetAndSubmitWorkdayReport`
- 問題: 登録、詳細取得、自動計算、編集、提出、提出後編集不可を1つのテストで検証している。失敗時にどの仕様が壊れたかを読み解くコストが高い。
- 推奨: `createWorkdayReportCalculatesMinutes`、`editDraftReportKeepsDraft`、`submitDraftReportChangesToPending`、`pendingReportCannotBeEdited` のように分割する。

### R-DR-008 `DailyReportResponse.from` のフィールド対応が長く、変更時に追いにくい

- 重要度: Low
- 対象: `backend/src/main/java/com/example/dailyreport/report/DailyReportResponse.java`
- 該当箇所: `DailyReportResponse.from`
- 問題: record の項目数が多く、コンストラクタ呼び出しの位置対応に依存している。項目追加・順序変更時のレビュー負荷が高い。
- 推奨: レスポンス項目をヘッダ、時間集計、監査情報、作業明細にグルーピングする、またはマッピング補助メソッドを分ける。

## 総評

既存ログイン機能と同じ薄い Controller/Service/Repository 構成に沿っており、初期実装としての一貫性はある。一方で、ログイン機能レビューで改善済みだった「責務分割」と「共通処理の切り出し」が、日報登録・編集の追加で再び目立っている。

優先度としては、まず `DailyReportForm` と `TimeRules.calculate` を分割し、次に時間帯の名前付き構造化、API 共通処理の統合、テスト分割を行うと、今後の一覧・承認・差戻し追加時の読みやすさが保ちやすい。

## 対応結果

| 指摘ID | 対応 |
| --- | --- |
| R-DR-001 | `DailyReportForm.tsx` を作成し、`App.tsx` から日報フォームを切り出した。作業明細 UI は `WorkItemsEditor`、状態管理は `useDailyReportEditor` に分割 |
| R-DR-002 | `TimeRules.calculate` を `validateAndCalculate`、休日区分別検証、勤務日検証、勤務時間計算へ分割 |
| R-DR-003 | `TimePeriod` record と名前付き時間帯定数を追加し、分単位の直値と `int[]` を削減 |
| R-DR-004 | `DailyReportService` で計算結果を1回だけ取得し、`apply` に渡す形へ変更 |
| R-DR-005 | `apiClient.ts` を追加し、JSON 読取、エラー変換、CSRF ヘッダー生成を共通化 |
| R-DR-006 | `MasterController` を追加し、マスタ API と日報 API を分離 |
| R-DR-007 | `DailyReportControllerTest` の正常フローを登録、自動計算、編集、提出、提出後編集不可に分割 |
| R-DR-008 | `WorkMinuteSummary` と `workItemResponses` を追加し、表示用時間と明細変換を補助メソッドへ分離 |
