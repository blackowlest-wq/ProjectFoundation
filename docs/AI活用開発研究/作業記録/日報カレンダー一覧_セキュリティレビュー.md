# 日報カレンダー・一覧画面 セキュリティレビュー

## 対象

- 日報カレンダー・一覧画面
- `GET /api/daily-reports`
- 関連する日報詳細 API、認証、共通 API client

## レビュー方法

- `セキュリティレビュー依頼` テンプレートに沿って確認した。
- `superpowers:dispatching-parallel-agents` を使い、認可/IDOR、入力検証/SQL/ログ、フロントセッション/CSRF/個人情報表示の3観点を並列確認した。
- Codex Security の `security-diff-scan` skill を確認したが、この環境では Git-backed 差分を解決できないため、フルスキャン成果物は生成せず、対象ファイル指定の手動レビューとして実施した。

## 問題なしと判断した観点

- `GET /api/daily-reports` は `SecurityConfig` の `anyRequest().authenticated()` により未ログインでは拒否される。
- 一覧APIはGETであり、CSRF対象外で妥当。登録、更新、提出、再提出、ログアウトはCSRF付き関数を使っている。
- 社員の一覧検索はバックエンドで `employeeUserId = 自分` に固定され、画面の社員ID入力だけで他者日報を取得する経路は成立しない。
- 上長の一覧検索は `manager_group_permissions` に基づき担当グループに限定される。
- SQL injection は確認範囲では成立しにくい。JPA Criteria API と `JdbcTemplate` プレースホルダを使っている。
- 対象差分内にログ出力はなく、パスワード、Cookie、セッションID、個人情報をログへ出す実装は確認されなかった。

## 残るリスク

- `DRL-R-002` は実装後レビュー側の仕様保留事項として残る。上長・管理者の同日複数社員日報をカレンダーでどう表現するかは、詳細導線または日別サマリ仕様の確定後に再確認する。
- Codex Security の Git-backed 差分スキャンは、この環境で Git ワークツリーを解決できず未実施。対象ファイル指定の手動レビューとOracle実機テストで補完した。

- `DR-S-003`: `DailyReportService#get` に社員本人、上長担当グループ、管理者全件の詳細参照認可を追加し、上長の担当外詳細参照が403になるMockMvcテストで確認した。
- `DRL-S-001`: 一覧検索期間を366日以内に制限し、超過時400になるMockMvcテストで確認した。
- `DRL-S-002`: 型変換例外を共通エラー形式に統一し、不正日付・不正承認状態のMockMvcテストで確認した。
- `DRL-S-003`: no content系API失敗時の例外化とログアウト失敗時の状態維持を、フロント単体テストで確認した。
- `DRL-S-004`: 一覧検索401時に一覧データを消してログイン画面へ戻ることをE2Eで確認した。
- `DRL-S-005`: 一覧・詳細の未ログイン、社員検索の他者ID無効化、CSRFヘッダー退行の回帰テストを追加した。
