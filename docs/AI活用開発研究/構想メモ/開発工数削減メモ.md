# AI活用開発 検討メモ: 開発工数の削減

## 基本方針

開発工数の削減は、AIあり開発だけを実施し、AIを使わない従来開発の見積もり工数と、AI活用時の実績工数を比較して評価する。

厳密なAIなし/AIあり比較ではなく、初期導入判断の参考値として扱う。
従来見積もりの精度に依存するため、削減率だけを絶対的な成果として扱わない。

また、工数削減だけで成功判定しない。
工数が削減されていても、品質指標が悪化している場合は効果ありとは判断しない。

## 比較方式

- 比較対象: AIを使わずに開発した場合の従来見積もり
- 実績: AIを使って実際に開発した工数
- 測定粒度: 機能単位
- 従来見積もり: 経験ベースで作成する

AI利用に伴う以下の時間も、実績工数に含める。

- AIへの指示作成時間
- AI生成結果の確認時間
- AI生成物の修正時間
- 再生成や追加指示の時間
- AI起因の手戻り対応時間
- レビュー指摘対応時間

## 対象機能

日報管理システムのうち、集計まで含めた以下の機能を対象にする。
詳細なスコープは `../サンプル設計書/スコープ定義.md` を参照する。

- ログイン
- 日報登録
- 日報編集
- 日報一覧・検索
- 日報詳細
- 日報承認
- 日報差戻し
- 未承認一覧
- 月次集計
- CSV出力

## 機能タイプごとの見方

### CRUD系

対象機能:

- 日報登録
- 日報編集
- 日報一覧・検索
- 日報詳細

主に、画面・API・DBの実装効率化が見やすい。

### 業務フロー系

対象機能:

- 日報承認
- 日報差戻し
- 未承認一覧

主に、状態遷移、権限、異常系の品質確認に向いている。

### 集計・出力系

対象機能:

- 月次集計
- CSV出力

主に、集計ロジック、境界値、出力形式のテストでAI効果を見やすい。
一方で、仕様が曖昧だとAIが誤った実装を生成しやすいため、事前に仕様を明確にする必要がある。

## 従来見積もりの前提

従来見積もりは経験ベースで作成する。
ただし、後から説明できるように、見積もり前提を残す。

見積もり条件案:

- Java / Spring Boot、React + TypeScript の基本知識がある開発者を想定する
- 基本設計・詳細設計は完了している前提とする
- 実装、単体テスト、APIテスト、画面確認を含む
- レビュー指摘対応を含む
- CI/CD構築などの基盤作業は機能別工数には含めない
- AIを使わない従来開発を想定する

## 主指標

- 機能単位の従来見積もり工数
- 機能単位のAI活用実績工数
- 工数差分
- 工数削減率

工数削減率は以下で計算する。

```text
工数削減率 = (従来見積もり工数 - AI活用実績工数) / 従来見積もり工数
```

## 副指標

- AI指示・対話時間
- AI生成物確認時間
- AI生成物修正時間
- 再生成回数
- AI起因の手戻り時間
- レビュー指摘対応時間
- 欠陥件数
- 重大度
- 欠陥分類
- 検出工程

AI生成コードやAI生成テストコードの採用率は参考値として扱う。
採用率が高くても品質が低い可能性があり、採用率が低くてもたたき台や発想補助として有効な場合があるため、主指標にはしない。

## 記録テンプレート

```text
機能名:
従来見積もり工数:
見積もり根拠:
AI活用実績工数:
AI利用に伴う確認・修正工数:
レビュー指摘対応工数:
工数差分:
工数削減率:
AIとの対話回数:
再生成回数:
AI起因の手戻り時間:
欠陥件数:
重大度:
欠陥分類:
検出工程:
判定:
```

## 判定基準

```text
効果あり:
  工数削減があり、品質指標が悪化していない

要注意:
  工数削減はあるが、欠陥や修正負荷が増えている

効果薄:
  工数削減が小さい、またはAI確認・修正工数が大きい
```

## 注意点

月次集計とCSV出力は、AIでコード生成しやすい一方、仕様の曖昧さが誤実装につながりやすい。

月次集計では、以下を事前に明確にする。

- 対象期間
- 承認済みだけを集計するか
- 差戻し中の日報を含めるか
- 休暇・未入力日をどう扱うか
- 作業時間の丸めをどうするか

CSV出力では、以下を事前に明確にする。

- 文字コード
- ヘッダー名
- 日付形式
- 改行・カンマ・引用符の扱い
- 出力対象の権限
