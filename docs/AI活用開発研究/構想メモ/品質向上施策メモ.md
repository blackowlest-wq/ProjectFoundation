# AI活用開発 検討メモ: 品質向上施策

## プロジェクト概要

AIを使った開発がまだ導入されていない開発現場を想定し、AI活用開発の研究・検討を行う。

開発モデルはウォーターフォールとし、まずは導入しやすい開発からテストまでの工程を対象にする。
小さなサンプル開発を実際にAIで回し、効果を検証する。

## サンプル開発の前提

- 題材: 社内向け日報管理システム
- 対象: 画面ありのWebアプリ全体
- フロントエンド: React + TypeScript
- バックエンド: Java / Spring Boot
- データベース: Oracle Database
- AI活用方針: 人間が設計し、AIが実装・テストを支援する
- スコープ定義: `../サンプル設計書/スコープ定義.md`

## 品質向上の基本方針

品質向上の目的は、AIの指摘件数を増やすことではなく、人間が確認し、再現可能と判断した有効欠陥や品質リスクを把握し、テストとレビューの補強につなげることとする。

AIレビューの指摘件数は主指標にしない。
AIのモデル更新、プロンプト差、ハルシネーション、仕様通りの実装に対する過剰指摘などにより、品質指標として安定しないため。

AIレビューは品質そのものを測る指標ではなく、欠陥候補を出す手段として扱う。

## 主指標候補

- 人間が確認し、再現可能と判断した有効欠陥数
- 重大度別の有効欠陥数
- 欠陥分類別の有効欠陥数
- 検出工程別の有効欠陥数

## 副指標候補

- AI指摘の有効率
- AI指摘の不採用率
- AI生成テストで検出された欠陥数
- 人間レビューで追加検出された欠陥数
- AIにより追加された有効テストケース数

## テスト観点の扱い

テスト観点の網羅性は、絶対的に測るのではなく、あらかじめ定義した観点分類に基づいて管理する。

AIには「テスト観点を出して」と漠然と依頼するのではなく、観点分類に沿って、不足している観点を洗い出させる。

観点分類、欠陥分類、重大度、検出工程、記録テンプレートは `メトリクスと分類.md` に集約する。

## 評価の考え方

AIが多く指摘したかではなく、AI活用により、確認可能な欠陥や品質リスクを発見し、テスト・レビューに反映できたかを見る。

品質向上の評価は、次の組み合わせで行う。

- 主: 有効欠陥の記録
- 主: 重大度、欠陥分類、検出工程による整理
- 副: AIによる有効欠陥候補の発見
- 副: AI生成テストによる欠陥検出
- 副: テスト観点分類に基づく補強
