# 共通部品化判断基準

## 目的

共通化の不足による重複と、過剰な共通化による複雑化の両方を防ぐ。
「似ているから共通化する」のではなく、同じ責務で同じ変更理由を持つかを基準に判断する。

## 共通化する

- 同じ責務で、変更理由も同じ処理が複数箇所にある。
- 認証、CSRF、API client、エラー応答、テストログイン補助など、誤ると影響が大きい処理。
- 仕様変更時に複数箇所を同時修正する必要が高い処理。
- 呼び出し側に不要な実装詳細が漏れている処理。
- テストやE2Eで同じ準備処理を繰り返している処理。

## 共通化しない

- 見た目や処理手順が似ているだけで、業務意味や変更理由が違う。
- 仕様がまだ揺れていて、抽象化すると変更しにくくなる。
- 1箇所だけの処理で、再利用見込みが低い。
- 共通化により引数や分岐が増え、呼び出し側より共通部品の方が複雑になる。
- 業務機能ごとの言葉や状態を、無理に汎用名へ変換する必要がある。

## 保留する

- 将来増える見込みはあるが、現時点で利用箇所が少ない。
- 既存実装との差分が大きく、別作業として安全に進めた方がよい。
- 共通化の責務境界に迷う場合。
- 共通化後の回帰確認コストが高く、今回の作業範囲を超える場合。

保留する場合は、保留理由、再確認タイミング、候補となる配置先を作業記録に残す。

## 優先的に共通化を検討する処理

| 領域 | 例 |
| --- | --- |
| フロントAPI | `fetch`、CSRFヘッダー、`credentials: 'include'`、エラー変換 |
| 認証 | ログイン状態確認、ログアウト、認証エラー処理 |
| バックエンドテスト | MockMvcログイン、CSRF付与、テストユーザー作成 |
| E2E | 認証モック、静的配信、共通操作、APIモック |
| エラー処理 | エラーコード、HTTPステータス、画面表示メッセージ |
| DB/マスタ | マスタ取得、状態判定、共通クエリ |

## 共通化後の確認

- 利用元の代表テストが通るか。
- 型チェック、ビルド、静的解析が通るか。
- 共通部品の名前が責務を表しているか。
- 共通部品が特定機能に依存しすぎていないか。
- 呼び出し側のコードが以前より読みやすくなっているか。
