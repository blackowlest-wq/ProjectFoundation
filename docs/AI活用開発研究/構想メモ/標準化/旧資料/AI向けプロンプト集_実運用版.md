# AI向けプロンプト集 実運用版

## 目的

AIへの依頼をフェーズ別に固定し、Planなし実装、テスト不足、レビュー記録漏れを減らす。

## 00 共通前提

```text
あなたは ProjectFoundation の開発支援AIです。
技術スタックは Spring Boot、React + TypeScript + Vite、Oracle Database、JUnit/MockMvc、Vitest、Playwright です。

必ず既存のログイン機能、共通API、DB接続、テスト方針、作業記録、指摘一覧に合わせてください。
作業前に AGENTS.md を確認し、作業内容に応じてチーム開発規約を参照してください。
仕様外の判断は実装せず、確認事項として分離してください。
軽微な仮定で進める場合は、仮定と理由を明記してください。
レビューで問題を検出した場合は、作業記録だけでなく指摘一覧にも記録してください。
ユーザから追加指摘を受けた場合は、規約・レビュー観点・チェック表へ反映すべきか確認してください。
```

## 01 Plan

```text
まだコード変更しないでください。

対象機能について、設計書と既存実装を読み、以下を整理してください。

- 実装対象
- 対象外
- 受入条件
- 確認したチーム開発規約
- BDDシナリオ
- TDD対象
- 必要なAPI
- 必要なDB/DDL/マスタ
- 必要なフロント画面
- 認証認可/CSRF/セッション
- 既存共通部品の利用箇所
- ファイル配置と命名方針
- 並列開発時のファイル衝突リスクと、ユースケース別ファイル分割方針
- 必要なテスト
- 静的解析/品質ゲート
- 作業記録テンプレートに記録する内容
- 確認事項
- 仮定
- 保留事項
```

## 02 BDDシナリオ作成

```text
受入条件からBDDシナリオを作成してください。

出力形式:
| No | Given | When | Then | 対応する受入条件 | テスト種別 |

注意:
- 仕様根拠があるものだけを確定シナリオにしてください。
- 仕様確認が必要なものは確認事項へ分けてください。
```

## 03 TDDテスト設計

```text
以下の対象について、先にテストケースを設計してください。

対象:
- 入力チェック
- 業務ルール
- 状態遷移
- 時刻/金額/集計などの計算
- 認可
- DB整合性

出力形式:
| No | 対象 | ケース | 前提 | 入力 | 期待結果 | 実装するテスト |
```

## 04 テストケースレビュー

```text
作成済みのテストケースをレビューしてください。

前提:
- AGENTS.md を確認してください。
- 単体テストケースレビュー観点.md、チーム開発_テスト方針.md、テスト静的解析チェック表.md を確認してください。
- 対象機能の設計書、受入条件、実装差分、既存テストを確認してください。
- 必要に応じて Serena MCP で既存テスト構造を確認してください。
- 独立した観点が複数ある場合は、superpowers:dispatching-parallel-agents の利用も検討してください。

対象:
{対象機能}

レビュー対象:
- テストケース資料
- 追加または変更された単体テスト
- API/Controllerテスト
- フロント単体テスト
- E2E
- Oracle/DB接続テスト
- 未実行テストと理由

観点:
- 受入条件、仕様、対象外事項との対応
- 正常系、異常系、境界値、同値分割
- ロール、権限、所有者、IDOR
- 入力チェック、状態遷移、DB整合性
- CSRF、セッション、未ログイン
- 0件、1件、複数件、大量件数
- 日付、月末、うるう年、タイムゾーン
- 回帰テスト、過去指摘、ユーザ追加指摘
- テスト名、テストデータ、モック妥当性
- Oracle実機確認、未実行テストの再確認条件
- 作業記録、指摘一覧、テストケース資料の更新漏れ

出力形式:
| 指摘ID | 重要度 | 分類 | 対象 | 内容 | 根拠 | 追加/修正すべきテスト | 対応要否 |

注意:
- 指摘がない場合も、確認した観点と残るリスクを書いてください。
- 問題を検出した場合は、作業記録だけでなく指摘一覧にも記録してください。
- 保留にする場合は、保留理由と再確認条件を書いてください。
```

## 05 Do

```text
Planとテスト設計に従って実装してください。

要求:
- 既存構成、命名、共通部品に合わせる。
- チーム開発_ディレクトリ構成ルール、セキュリティ規約、共通部品化判断基準に合わせる。
- 別機能の変更が同一ファイルに集中する場合は、Controller、Service、Repository、テストをユースケース別に分離する。
- 機能変更と大きなリファクタを混ぜすぎない。
- バックエンド入力チェックを正とし、フロント入力チェックは利用者補助とする。
- Oracle前提でDB定義とテストを考える。
- セキュリティ、認証認可、CSRFを崩さない。
- ファイル冒頭コメントと、意味のある分岐・難しい処理のコメントを入れる。
- 変更後に必要なテストを実行し、結果を記録する。
```

```text
- 並列開発時のファイル衝突リスク

- コメント

- カバレッジ

問題があれば指摘一覧へ記録し、修正してください。
```

## 07 Fix

```text
以下の指摘を修正してください。

要求:
- 指摘ごとに原因、修正内容、確認方法を整理する。
- 修正に対応するテストを追加または更新する。
- 修正後に必要なテストを再実行する。
- 指摘一覧と作業記録を更新する。
- ユーザからの追加指摘が含まれる場合は、規約・レビュー観点・チェック表へ反映すべきか判断し、反映または保留理由を記録する。
```

## 08 Record

```text
作業記録と指摘一覧を更新してください。

作業記録に含める内容:
- 実装工数
- テスト作成工数
- 環境構築・DB・テストデータ準備工数
- 確認した規約・チェック表
- 手戻り内容
- 発生したエラー
- AI出力を人間が修正した内容
- 採用判断
- 実装時の仮定
- 未対応・保留事項
- 実行したテストと結果
- 未実行テストの理由と再確認条件
- ユーザ追加指摘の規約反映有無

指摘一覧に含める内容:
- 指摘ID
- 分類
- 重要度
- 指摘内容
- 対応状況
- 対応内容または保留理由
```
