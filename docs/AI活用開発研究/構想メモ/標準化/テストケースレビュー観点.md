# テストケースレビュー観点

## 目的

単体テスト、API/Controllerテスト、フロント単体テスト、E2E、Oracle/DB接続テストをレビューするときに、仕様担保、境界値、実装分岐、品質補完の抜け漏れを確認するための共通チェックリストを定義する。

## 使い方

1. 対象機能の受入条件、入力チェック、業務ルール、非機能要件を確認する。
2. 既存テストケースを下記チェックリストに対応付ける。
3. 判定は `OK`、`不足`、`対象外`、`保留` のいずれかにする。
4. 受入条件ID、過去障害・指摘ID、テストケースID、実テストIDの対応を確認する。
5. 期待結果が仕様根拠、観測可能性、十分なアサーションを持つかを独立に確認する。
6. `不足` は追加候補として記録し、仕様上必要か、カバレッジ目的か、回帰防止目的かを分類する。

## 基本観点

| 観点 | 確認内容 |
| --- | --- |
| ブラックボックス | 仕様書、API一覧、受入条件どおりの入力と出力を確認しているか |
| ホワイトボックス | 実装上の重要な分岐、例外、状態変化を確認しているか |
| 正常系 | 代表的な成功パターンを確認しているか |
| 異常系 | 不正入力、未認証、権限不足、存在しないデータを確認しているか |
| 最小値 | 入力可能な最小値、最小件数を確認しているか |
| 最大値 | 許容最大値、最大文字数、上限件数を確認しているか |
| 境界値 | 境界値そのものと境界の外側を確認しているか |
| 同値分割 | 同じ扱いになる入力群から代表値を選んでいるか |
| 区分値・Enum網羅 | 定義済み区分を全件確認しているか。未定義値を拒否しているか |
| 組み合わせ | ロール、状態、区分、入力有無などの組み合わせを確認しているか |
| デシジョンテーブル | 条件が多い業務ルールを表にして漏れなく確認しているか |
| 状態遷移 | 正常遷移、不正遷移、遷移後の副作用を確認しているか |
| 状態別画面操作 | 状態ごとに入力、明細追加・更新・削除、保存、提出の許可/拒否を一覧化しているか |
| 読込中・読込失敗 | 詳細取得中・取得失敗時に初期値の誤保存や状態不明の操作が発生しないことを確認しているか |
| null・未指定 | null、未指定、空配列などを確認しているか |
| 空文字・空白 | 空文字、半角空白、全角空白を確認しているか |
| 形式チェック | 日付、時刻、年月、コード形式などを確認しているか |
| 桁数・精度 | 文字数、数値桁、丸め、分単位などを確認しているか |
| 重複・一意制約 | 同じキーや業務上一意のデータを拒否できるか |
| 存在チェック | 参照先データや対象データが存在しない場合を確認しているか |
| 権限・所有者 | 自分、他人、上長、管理者など権限差を確認しているか |
| 順序・並び替え | 一覧や検索結果の順序を確認しているか |
| 件数 | 0件、1件、複数件、大量件数を確認しているか |
| 複数データ読み込み | 複数レコードや複数ロールの読み込みで期待どおりか |
| N+1 | データ件数増加時にSQLや処理回数が不自然に増えないか |
| 排他・同時実行 | 同時更新、古い状態、二重実行を確認しているか |
| 副作用 | DB更新、セッション、ログ、イベントなどの副作用を確認しているか |
| 冪等性 | 同じ操作を複数回実行しても破綻しないか |
| タイムゾーン・日付境界 | 月末、年末、日跨ぎ、タイムゾーン差異を確認しているか |
| ロケール・文字種 | 日本語、全角、記号、改行、絵文字などを確認しているか |
| セキュリティ | 認証、認可、CSRF、XSS、SQLインジェクションを考慮しているか |
| エラー処理 | 期待するステータス、エラーコード、メッセージ、詳細を返すか |
| DB依存 | 実DB、方言、制約、型差異を確認しているか |
| 回帰テスト | 過去の不具合やレビュー指摘の再発を防ぐテストがあるか |
| カバレッジ補完 | 未通過分岐を意図して確認しているか。仕様担保との違いを明記しているか |

## テスト構成・責務分割観点

テストが成功することと、テスト構成が保守可能で漏れを検出しやすいことは別に判定する。実装前と実装後、Full実行前に次を確認する。

| 観点 | 確認内容 |
| --- | --- |
| 責務境界 | テストファイルが機能、ユースケース、コンポーネントの境界に沿い、認証・一覧・登録・編集など独立した責務を不必要に混在させていないか |
| テスト層境界 | Unit/UI/API/E2E/Oracleの目的と実行環境が一つのファイルへ不必要に混在していないか |
| ファイル集中 | 一つのテストファイルに複数責務のdescribe、fixture、モックが集中し、変更時の影響範囲やレビュー単位を不明確にしていないか |
| 分割・維持判断 | 分割する場合は対象ファイル、責務、移動範囲、維持する期待値を明示し、分割しない場合は理由と再確認条件を記録しているか |
| 移動の同値性 | 分割・移動でテスト名、アサーション、fixture、モック応答、ケースID、実テストIDが意図せず変わっていないか |
| 共通support | 複数ファイルで共有する準備・fixture・操作helperがsupportへ整理され、特定の代表テストだけに隠れていないか |
| トレーサビリティ | テスト構成・責務対応表から正本ケース、実テストファイル、テスト名、実行コマンドを追跡できるか |
| Fullとの区別 | Full成功を構成レビューの証拠とせず、構成レビューの判定とFull結果を別々に記録しているか |

## 区分値・Enumがある場合の重点観点

| 観点 | 確認内容 |
| --- | --- |
| 定義済み値の全件確認 | Enumや区分値を全件1回以上確認しているか |
| 未定義値の拒否 | 存在しない区分値、空、nullを拒否しているか |
| 区分ごとの業務差分 | 区分によって結果や権限が変わる箇所を確認しているか |
| 区分と状態の組み合わせ | 区分 × 状態 × 操作の許可/拒否を確認しているか |
| 追加時の変更容易性 | Enum追加時にテストが不足を検知できるか |

## トレーサビリティ観点

| 観点 | 確認内容 |
| --- | --- |
| 受入条件との紐づけ | 各テストケースに受入条件IDがあり、未対応の受入条件が対象外・保留を含めて明示されているか |
| ID名前空間 | 受入条件、テストケース、実テスト、指摘のIDを別名前空間で管理し、同じ文字列を再利用していないか |
| 過去障害との紐づけ | 過去障害・レビュー指摘に関連する回帰テストがあり、欠陥IDとテストケースIDが相互参照できるか |
| 実テストとの紐づけ | テストケースからテストメソッド、spec、実行結果、証跡を追跡できるか |
| 孤立ケース | 要求・障害・探索目的のいずれにも紐づかないケースがないか |
| 重複・矛盾 | 同じ目的の重複ケースや、期待結果が矛盾するケースがないか |

## 期待結果レビュー観点

| 観点 | 確認内容 |
| --- | --- |
| 仕様根拠 | 期待結果が受入条件、設計、過去障害のいずれかに根拠づけられているか |
| 観測可能性 | API本文、画面、DB、セッション、イベント、ファイルなど確認対象が具体的か |
| 検出力 | 仕様違反や条件反転が発生した場合にテストが失敗するアサーションか |
| 副作用 | DB更新、状態変更、ログ、通知、セッションなど必要な副作用を確認しているか |
| テスト層適合 | 単体、API、画面、E2E、Oracleの適切な層で確認しているか |
| 曖昧さ | 仕様不明点を推測で期待結果にせず、確認事項・保留としているか |

## 過去障害・回帰観点

| 観点 | 確認内容 |
| --- | --- |
| 未検出理由 | 既存テストで検出できなかった理由が、ケース不足、期待結果不足、データ不足などに分類されているか |
| 回帰テスト | 修正後に同じ障害を検出できるテストが追加・修正されているか |
| 横展開 | 同じ原因が他機能にも存在しないか確認しているか |
| 標準化反映 | 受入条件、ケース様式、レビュー観点、プロンプト、サンプルの必要な箇所へ反映されているか |

## 専門レビュー結果

|レビュー担当|入力資料|判定|指摘ID|標準化判定・反映先|未確認・保留理由|
|---|---|---|---|---|---|
|要求照合|受入条件、設計書、ケース|||||
|過去障害・回帰|欠陥記録、ケース、実テスト|||||
|敵対的・セキュリティ|ケース、API、認証認可、状態|||||
|状態・境界値・組合せ|状態遷移、入力規則、テストデータ|||||
|期待結果・カバレッジ|ケース、実装、アサーション、カバレッジ|||||
|トレーサビリティ・統合|受入条件、ケース、実テスト、各専門レビュー|||||

## レビュー結果の記録形式

| 観点 | 判定 | 現行テスト | 不足・追加候補 | 根拠ID | 期待結果レビュー | 専門レビュー | 優先度 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 例: 境界値 | 不足 | 81文字ログインIDのみ | 80文字OK、100文字パスワードOK、101文字パスワードNG | F-001-AC-006 | 要修正 | 状態・境界値・組合せ | Medium |
