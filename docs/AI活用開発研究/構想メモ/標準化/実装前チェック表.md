# 実装前チェック表

## 目的

AIに実装を依頼する前に、仕様・既存実装・テスト方針・記録方針を揃える。

## 使い方

- 実装前のPlanフェーズで確認する。
- この資料は、作業ごとに確認結果を埋めるチェック表として使う。
- 開発標準の判断基準は `実装前確認観点.md` を参照する。
- 不明点は「確認事項」として分離する。
- 軽微な仮定で進める場合は、仮定と理由を作業記録へ残す。

## チェック表

| 分類 | チェック内容 | 確認結果 | 備考 |
| --- | --- | --- | --- |
| 依頼範囲 | 対象機能、対象外、今回の完了条件が明確か |  |  |
| 設計書 | スコープ、機能一覧、画面、API、DB、入力チェック、状態遷移、非機能、テスト設計を確認したか |  |  |
| 受入条件 | 受入条件をケース単位に分解したか |  |  |
| BDD | 主要な業務フローをGiven/When/Thenで表現できるか |  |  |
| TDD | 入力チェック、業務ルール、状態遷移、計算ロジックのテストを先に考えたか |  |  |
| 既存実装 | ログイン機能、共通API、CSRF、セッション、DB接続、テスト方針を確認したか |  |  |
| 共通化 | 既存共通部品を使うべき箇所を確認したか |  |  |
| DB | Oracle前提のDDL、マスタ、テストデータ、接続方法を確認したか |  |  |
| 認証認可 | 未ログイン、他者データ、ロール別操作の扱いを確認したか |  |  |
| セキュリティ | CSRF、セッション固定、機密情報露出、IDORを確認したか |  |  |
| ファイル配置 | feature別、controller/entity/service/dto/logicなど配置方針を確認したか |  |  |
| 命名 | Entity、DTO、Repository、API関数、画面部品の命名方針を確認したか |  |  |
| 成果物の役割 | 本体コードはHow、テストコードはWhat、コミットログはWhy、コードコメントは必要なHowとWhy notとする方針を確認したか |  |  |
| コメント | 必要な責務・境界の冒頭コメントと、分岐・重要処理の`How:`を残し、採用しなかった方式は`Why not:`で説明する方針を確認したか |  |  |
| テスト表現 | テスト名・`describe`/`it`からWhatと期待結果が分かる方針を確認したか |  |  |
| コミット・記録 | コミット本文のWhyと、Why notの判断・不採用方式の作業記録先を確認したか |  |  |
| 静的解析 | lint、format、型チェック、静的解析の実行有無を確認したか |  |  |
| テスト | 単体、結合寄り、E2E、カバレッジ、Oracle実機確認の範囲を決めたか |  |  |
| 記録 | 作業記録、指摘一覧、レビュー記録、テストケース資料の更新先を決めたか |  |  |
