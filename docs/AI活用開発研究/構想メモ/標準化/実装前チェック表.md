# 実装前チェック表

## 目的

AIに実装を依頼する前に、仕様・既存実装・テスト方針・記録方針を揃える。この資料は、作業ごとに結果を埋めるチェック表と、実装前のチーム共通判断基準を兼ねる。

## 使い方

- 実装前のPlanフェーズで確認する。
- この資料は、作業ごとに確認結果を埋めるチェック表として使う。
- 不明点は「確認事項」として分離する。
- 軽微な仮定で進める場合は、仮定と理由を作業記録へ残す。
- チェック結果を埋めるだけでなく、今回の対象範囲、適用するP0/P1/P2観点、対象外・保留理由を記録する。
- テストを追加・変更する場合は、実装前にテスト構成・責務対応表を作り、既存テストの分割・維持判断とテスト層を確定する。全テスト成功はこの確認の代わりにしない。

| テスト分離方針 | 既存テストを分離する場合、期待値・アサーション・テストケースを維持し、import/helper/クラス配置だけを変更する計画か | | |
| テスト構成・責務分割 | 既存・追加テストを機能、ユースケース、コンポーネント、テスト層ごとに棚卸しし、独立した責務を同一テストファイルへ集中させない計画か。分割しない場合の理由を記録したか | | |
| 命名 | Entity、DTO、Repository、API関数、画面部品の命名方針を確認したか | | |
| 成果物の役割 | 本体コードはHow、テストコードはWhat、コミットログはWhy、コードコメントは必要なHowとWhy notとする方針を確認したか | | |

| 最終ゲート | 専門レビューと指摘修正後に、Full、E2E、coverage、Oracleなどの最終確認を行う順序を決めたか | | |

## 適用する確認観点の優先度

| 優先度 | 適用内容 |
| --- | --- |
| P0 | 対象範囲、設計・受入条件、正本ケース、期待結果、トレーサビリティ、テスト構成・責務境界、外部ドキュメント、TDD/BDD、最終ゲートを確認する。 |
| P1 | 認証・認可・CSRF・入力検証、配置・共通化、静的解析・CI、ログ・観測可能性を変更内容に応じて確認する。ログは本資料の「ログ設計」と`セキュリティ規約.md`を主資料とする。 |
| P2 | 性能、競合、互換性、アクセシビリティ、移行・復旧などを明示要件または変更リスクがある場合に確認する。 |

P1/P2を適用しない場合も、理由または保留条件を作業記録へ残す。
