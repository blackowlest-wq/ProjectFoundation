# 実装前確認観点

## 目的

実装前に目的、対象範囲、既存実装、外部ドキュメント、テスト方針を揃え、手戻りと判断漏れを減らす。

## 基本方針

- コーディング前に確認し、未確認事項を分離する。
- この資料は、開発標準の判断基準として使う。
- 作業ごとの確認結果を埋める場合は、既存の `実装前チェック表.md` を併用する。
- 軽微な仮定で進める場合も、仮定と理由を作業記録に残す。
- 機能追加や大きめの修正では、既存の `実装前チェック表.md` も併用する。

## チェック項目

| 分類 | 確認内容 |
| --- | --- |
| 対象範囲 | 対象機能、対象外、完了条件が明確か |
| 設計書 | 画面、API、DB、入力チェック、状態遷移、非機能を確認したか |
| 受入条件 | 受入条件をケース単位に分解したか |
| 外部ドキュメント | 利用するライブラリ、フレームワーク、API、ツールの公式ドキュメントや一次情報を確認したか |
| 既存実装 | ログイン、共通API、CSRF、セッション、DB接続、テスト方針を確認したか |
| ログ設計 | 正常開始・完了、業務エラー、認証/認可拒否、システム失敗、外部I/Oなど、追跡に必要なイベント・終了状態・項目・相関IDを定義したか。ログレベル、出力量、保持/検索、マスキング、監視、テスト方法を決めたか |
| 共通化 | 既存共通部品を使うべき箇所を確認したか |
| DB | Oracle前提のDDL、マスタ、テストデータ、接続方法を確認したか |
| 認証認可 | 未ログイン、他者データ、ロール別操作の扱いを確認したか |
| セキュリティ | CSRF、セッション固定、機密情報露出、IDORを確認したか |
| ファイル配置 | feature別、controller/entity/service/dto/logicなど配置方針を確認したか |
| 並列開発 | 既存ファイルへ追記する場合、別機能との同時更新が集中しないか。必要ならユースケース別Controller/Service/Repository/テストへ分割する判断をしたか |
| テスト配置 | 本体配下へテストファイルを置かず、フロントは `frontend/test`、E2E は `frontend/e2e`、バックエンドは `backend/src/test` に配置する方針か |
| ユースケース分割 | Controller、Service、テストがユースケース単位に分かれ、複数機能を同一ファイルへ集中させない計画か |
| テスト分離方針 | 既存テストを分離する場合、期待値・アサーション・テストケースを維持し、import / helper / クラス配置だけを変更する計画か |
| 成果物の役割 | 本体コードはHow、テストコードはWhat、コミットログはWhy、コードコメントは必要なHowとWhy notとする計画か |
| コメント設計 | ファイル冒頭に責務・境界・契約が必要か、各メソッド・関数の責務・契約を示す日本語コメントが必要か、分岐・重要処理に誤読防止の`How:`が必要か、採用しなかった方式に`Why not:`が必要かを確認したか |
| テスト表現 | テスト名・`describe`/`it`・Arrange/Act/Assertから確認対象と期待結果が読める計画か |
| コミット・記録 | コミット本文にWhyを記載し、Why notの判断と不採用方式を作業記録へ残す計画か |
| テスト | 単体、結合寄り、E2E、Oracle実機確認、静的解析の範囲を決めたか |
| 記録 | 作業記録、指摘一覧、レビュー記録、テストケース資料の更新先を決めたか |
