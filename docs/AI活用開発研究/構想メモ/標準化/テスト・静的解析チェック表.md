# テスト・静的解析チェック表

## 目的

AI実装後に、テストと静的解析の確認漏れを防ぐ。

## テスト分類

| 分類 | 目的 | 代表コマンド/確認 |
| --- | --- | --- |
| バックエンド通常テスト | Controller、Service、業務ルール、認証を確認する | リポジトリルートから `backend/scripts/test-oracle.cmd -q test` |
| Oracle実機確認 | Oracle前提のDDL、SQL、接続、制約を確認する | `DAILY_REPORT_DB_*` を指定して実行 |
| JaCoCo | 未通過分岐とカバレッジを確認する | リポジトリルートから `backend/scripts/test-oracle.cmd -q -Pcoverage test` |
| フロント単体テスト | APIクライアント、入力チェック、画面部品を確認する | `npm.cmd test -- --run` |
| TypeScript型チェック | 型不整合、import漏れを確認する | `npm.cmd run typecheck` |
| フロントビルド | production buildで破綻しないか確認する | `npm.cmd run build` |
| Vitest coverage | 未通過のUI/ロジックを確認する | `npm.cmd run coverage` |
| E2E | 主要なユーザー操作フローを確認する | `npm.cmd run e2e` |

## 静的解析導入候補

| 領域 | 候補 | 導入目的 | 優先度 |
| --- | --- | --- | --- |
| Frontend | ESLint / typescript-eslint | 未使用、危険なReact hooks、型周辺のミス検出 | 高 |
| Frontend | Prettier | フォーマット差分の排除 | 中 |
| Backend | Spotless | Javaフォーマット統一 | 中 |
| Backend | Checkstyle | 命名、import、コメント、複雑度の規約化 | 中 |
| Backend | SpotBugs | Null、リソース、バグパターン検出 | 中 |
| Security | CodeQL | セキュリティ/品質の静的解析 | 中 |
| Security | secret scanning | パスワード、トークン、接続情報の混入防止 | 高 |
| Dependency | Dependabot / dependency check | 脆弱な依存関係の検知 | 中 |

## 品質ゲート案

- すべての通常テストが成功している。
- TypeScript型チェックが成功している。
- フロントビルドが成功している。
- E2Eが成功している。
- カバレッジが前回から大きく悪化していない。
- 静的解析を導入済みの場合、重大以上の指摘が残っていない。
- 未実行の確認がある場合は、理由と再実行条件を作業記録に残す。
- バックエンドテストは `backend/config/oracle-test.properties` を読み込む共通入口から実行する。直接 `mvn.cmd` を呼ぶと接続情報がプロセス環境変数へ渡らず、Oracle接続に失敗する。
