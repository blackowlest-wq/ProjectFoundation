# AI利用ルール・注意事項

## 目的

AI利用時のリスクを抑え、開発者ごとの使い方のバラつきを減らす。

本ルールは、社内向け日報管理システムのサンプル開発における開発からテストまでの工程を対象とする。

## AI利用の基本姿勢

- AIは開発者を支援する補助者として扱う
- 設計判断、仕様判断、採用判断は人間が行う
- AI生成物をそのまま信用しない
- AIの出力は、仕様、コード、テストで確認してから採用する
- 不明点は推測で埋めず、確認事項として扱う

## 禁止事項

- 機密情報、個人情報、顧客情報をAIに入力しない
- 未公開の社内資料や顧客資料を、許可なくAIに入力しない
- AI生成コードを人間レビューなしで採用しない
- AIが提示した仕様外の判断を、そのまま実装しない
- AIレビューの指摘件数を品質向上の主指標にしない
- AIが生成したテストを、内容確認せず網羅済みとして扱わない
- エラーや警告を理解せず、AIの修正案だけで解消しようとしない

## 入力情報の扱い

AIに入力する情報は、必要最小限にする。

入力してよい情報:

- サンプル開発用に作成した仕様
- サンプル開発用のコード
- 技術スタック
- 画面項目
- API仕様
- DB定義
- テスト観点
- 再現用のエラーメッセージ

入力を避ける情報:

- 実案件の機密情報
- 顧客名、社員名、メールアドレスなどの個人情報
- 社内限定の認証情報
- APIキー、パスワード、トークン
- 本番データ

## 仕様不明点の扱い

AIが仕様にない内容を補完した場合は、実装せず確認事項として扱う。

確認事項にする例:

- 承認済み日報を編集できるか
- 差戻しコメントを必須にするか
- 月次集計に未承認日報を含めるか
- CSVの文字コードを何にするか
- 作業時間の丸め単位をどうするか

## ハルシネーション確認ルール

AI出力について、以下を確認する。

- 存在しないファイル、クラス、関数を参照していないか
- 存在しない仕様を前提にしていないか
- ライブラリの使い方が実際のバージョンと合っているか
- API名、テーブル名、項目名が正しいか
- エラーの原因を根拠なく断定していないか

疑わしい場合は、AI出力を採用せず、コード、公式ドキュメント、テスト結果で確認する。

## 採用・不採用の判断基準

採用できるもの:

- 仕様に合っている
- 既存設計と整合している
- テストで確認できる
- レビュー観点を満たしている
- 保守しやすい

修正して採用するもの:

- 方針は妥当だが命名や構成が合わない
- 正常系は合っているが異常系が不足している
- コードは使えるがテストが不足している
- 仕様確認が必要な箇所を除けば有効

不採用にするもの:

- 仕様と矛盾している
- 既存設計を大きく崩す
- 根拠のない仕様追加を含む
- セキュリティや権限制御に問題がある
- 修正量が多く、作り直した方が早い

## 重点確認領域

以下はAI生成物を特に慎重に確認する。

- 認証・認可
- 権限ごとの操作制御
- 承認・差戻しの状態遷移
- DB更新
- トランザクション
- 月次集計
- CSV出力
- 例外処理
- 入力チェック

## AI利用ログ

AIを使った場合は、最低限以下を残す。

```text
日付:
対象機能:
利用工程:
AIへの依頼内容:
AI出力の概要:
採用有無:
修正内容:
不採用理由:
確認者:
```

## 品質評価との関係

AIレビューの指摘件数は品質指標として扱わない。
AIの指摘は欠陥候補として扱い、人間が仕様根拠や再現性を確認したものだけ有効欠陥として記録する。

工数削減を評価する場合も、AIへの指示、確認、修正、再生成、手戻りの時間を実績工数に含める。

工数が削減されていても、欠陥件数、重大度、欠陥分類、検出工程の記録から品質面の問題が大きいと判断される場合は、効果ありとは判断しない。
