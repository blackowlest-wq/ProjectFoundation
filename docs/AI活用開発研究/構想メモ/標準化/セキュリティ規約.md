# セキュリティ規約

## 目的

認証、認可、入力検証、CSRF、ログ出力などの判断を揃え、機能追加時のセキュリティ退行を防ぐ。

## 認証・認可

- 認証が必要なAPIは、バックエンドで必ずログイン状態を確認する。
- 所有者やロールに依存する操作は、画面制御だけでなくバックエンドで認可する。
- IDを受け取るAPIでは、他者データを参照・更新できないことを確認する。
- 管理者、一般ユーザー、未ログインの動作を分けて確認する。

## CSRF・セッション

- POST、PUT、PATCH、DELETE などの変更系APIはCSRF対策を必須とする。
- ログイン成功後、ログアウト後、セッション期限切れの挙動を確認する。
- セッションID、CSRF token、Cookie値をログや画面に出さない。
- セッション状態をフロントだけで判断せず、バックエンドの結果を正とする。

## 入力検証

- フロントの入力制御は補助とし、バックエンドの検証を正とする。
- 文字数、必須、形式、日付範囲、数値範囲、状態遷移を検証する。
- 業務上存在しないID、他者データID、不正な状態遷移を拒否する。
- エラー応答は、利用者が修正できる情報に限定し、内部実装やSQL詳細を返さない。

## ログ・機密情報

- パスワード、トークン、Cookie、セッションID、個人情報、接続文字列をログに出さない。
- 調査に必要な場合は、IDや分類など最小限の情報にする。
- 例外ログは原因調査に必要な型・スタック位置の粒度を保ちつつ、例外メッセージや機密値を含めない。

## ログ相関・出力レベル

- Backendの各HTTPリクエストは相関IDを生成し、MDCと`X-Request-Id`レスポンスヘッダーへ同じ値を設定する。エラーJSONにも同じ`requestId`を含める。
- リクエスト完了ログには、相関ID、HTTPメソッド、クエリを除いたパス、機能、ユースケース、HTTPステータス、処理時間を含める。
- 業務エラー・認証拒否は`WARN`、予期しないシステム例外は`ERROR`、重要な状態変更は`INFO`、リクエスト開始・完了の詳細は通常`DEBUG`とする。E2Eや一時調査で必要な場合だけプロファイルまたはlogger levelを変更する。
- 業務エラーのログにはエラーコード、ステータス、詳細件数などを残すが、入力値やエラー詳細本文をそのまま記録しない。
- FrontendはBackendの`requestId`を`ApiError`へ保持し、500以上またはネットワーク障害だけをパス・ステータス・コード・相関IDに限定して診断出力する。リクエスト本文、ヘッダー、Cookieは出力しない。
- 実Backend/Oracle E2Eはcleanup前にDB名・サービス名・セッションユーザーを照合し、専用seed・固定cleanup・Oracle保存確認を行う。実ログインを含むE2E成果物ではPlaywrightのtrace/video/screenshotを保存しない。

## レビュー必須観点

| 観点 | 確認内容 |
| --- | --- |
| 未ログイン | 認証必須APIが拒否されるか |
| 他者データ | IDORが発生しないか |
| 権限 | ロール外操作が拒否されるか |
| CSRF | 変更系APIでCSRFが必要か |
| セッション | ログアウト後に操作できないか |
| エラー | 内部情報を返していないか |
| ログ | 機密情報を出していないか、相関ID・機能・ユースケース・ステータス・処理時間・レベル制御があるか |
