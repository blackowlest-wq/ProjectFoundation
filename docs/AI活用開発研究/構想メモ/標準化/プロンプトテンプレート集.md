# プロンプトテンプレート集

## 目的

AIへの指示のばらつきを減らし、開発者が同じ形式でAIを活用できるようにする。
各テンプレートは、必要に応じて対象機能、仕様、コード、制約を差し替えて使用する。

## 使い方

- 実装やレビューを依頼する前に、`AGENTS.md` と `標準化資料一覧.md` を確認する。
- 日本語資料を読む場合は UTF-8 として読み込む。
- 外部ライブラリ、フレームワーク、API、ツールを扱う場合は、実装判断の前に公式ドキュメントや一次情報を確認する。
- 構想、仕様整理、設計、実装計画が必要な場合は、superpowers の該当skillを使う。
- 既存コードの構造、参照関係、影響範囲を調べる場合は、Serena MCPを使う。
- セキュリティレビュー、脆弱性調査、攻撃経路確認が必要な場合は、Codex Securityを使う。
- レビューや独立調査で確認観点が複数ある場合は、superpowers のサブエージェント活用を検討する。
- 実装やレビュー後は、作業記録テンプレートに沿って記録する。

## ツール利用の目安

| 目的 | 優先して使うもの | 備考 |
| --- | --- | --- |
| 外部仕様確認 | 公式ドキュメント/一次情報 | React、Spring、Vite、Playwrightなどの公式ドキュメント確認 |
| 仕様整理・設計 | superpowers:brainstorming | 実装前に目的、制約、案を整理する |
| 実装計画 | superpowers:writing-plans | 承認済み設計を実装タスクに分解する |
| 並列レビュー・独立調査 | superpowers:dispatching-parallel-agents | 仕様、API、UI、DB、テスト、記録など独立した観点を並列確認 |
| コードレビュー依頼 | superpowers:requesting-code-review | 実装完了後、主要機能完了後、マージ前に第三者レビューを依頼 |
| レビュー指摘対応 | superpowers:receiving-code-review | レビュー指摘を受けた後、修正前に妥当性確認と対応順を整理 |
| TDD | superpowers:test-driven-development | 業務ルール、入力検証、状態遷移の実装前テスト設計 |
| 不具合調査 | superpowers:systematic-debugging | 再現、原因候補、検証、修正の順で進める |
| 完了確認 | superpowers:verification-before-completion | 完了宣言前にテスト結果を確認する |
| 既存コード調査 | Serena MCP | 定義、参照、シンボル、影響範囲を調べる |
| セキュリティ確認 | Codex Security | 認証認可、CSRF、IDOR、機密情報漏えいなど |

## 共通の依頼形式

```text
あなたは ProjectFoundation の開発支援AIです。
対象は社内向け日報管理システムです。

技術スタック:
- フロントエンド: React + TypeScript + Vite
- バックエンド: Java / Spring Boot
- データベース: Oracle Database
- テスト: JUnit/MockMvc、Vitest、Playwright

前提:
- AGENTS.md を最初に確認してください。
- 日本語ドキュメントは UTF-8 として読んでください。
- 作業内容に応じて開発標準資料を確認してください。
- 外部ライブラリ、フレームワーク、API、ツールを扱う場合は公式ドキュメントや一次情報を確認してください。
- 既存コードの構造や影響範囲を確認する場合は Serena MCP を使ってください。
- セキュリティ観点がある場合は Codex Security の利用を検討してください。
- 仕様外の判断は実装せず、確認事項として提示してください。
- 不明点がある場合は、推測で進めず確認事項にしてください。
- レビューで問題を検出した場合は、作業記録だけでなく指摘一覧にも記録してください。

対象機能:
{対象機能}

入力情報:
{仕様、API、DB定義、画面項目、関連コードなど}

制約:
{コーディング規約、禁止事項、既存設計、テスト方針など}
```
