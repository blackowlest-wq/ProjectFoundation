# 実装後レビュー表

## 目的

AIが実装した成果物を、人間が確認しやすい単位でレビューする。この資料は、作業ごとに結果を埋めるレビュー表と、実装後のチーム共通レビュー観点を兼ねる。

## 基本方針

- 差分の見た目だけでなく、変更理由と影響範囲を確認する。
- 仕様、責務、セキュリティ、テスト、記録の順に確認する。
- 正本テストケースと受入条件レビューが今回の変更を反映しているかを、コードレビューより先に確認する。
- 状態を持つ画面では、状態ごとの全操作と、詳細読込中・読込失敗・状態競合時の安全側制御を確認する。
- UIの編集可否判定と保存処理の拒否判定が同じ契約を使い、UI無効化を回避しても誤更新しないことを確認する。
- Full実行の成功はテスト構成の十分性を示さない。Full実行前に、テストファイルの責務境界、テスト層、配置、重複・混在を確認し、分割要否を判定する。
- 問題を検出した場合は、作業記録だけでなく指摘一覧にも残す。修正しない場合は、保留理由と再確認条件を明記する。
- ユーザー追加指摘は、今後も再発し得る観点かを判定し、必要なら規約・チェック表へ反映する。

## 詳細レビュー観点

### ロガーレビューの共通観点

対象システムに該当する項目を選び、ログが出ていることだけでなく、調査に使える契約になっていることを確認する。

- イベント網羅：正常開始・完了、業務エラー、認証/認可拒否、入力エラー、システム失敗、外部サービス/DB通信、非同期処理を必要な境界で記録しているか。
- 追跡可能性：requestId/相関IDがFrontend、Backend、外部呼出し、非同期処理、エラー応答まで伝播しているか。
- 結果の正確性：HTTP結果、業務結果、状態変更、処理時間が実際の終了結果と一致しているか。
- レベルと負荷：ERROR/WARN/INFO/DEBUGの意味、本番ログ量、保持期間、ローテーション、監視が定義されているか。
- 機密性：パスワード、トークン、Cookie、セッション情報、個人情報、入力本文、SQL、接続情報を出していないか。
- 構造化と契約：イベント名、項目名、エラーコード、レベルが安定しているか。
- エラー終端：業務エラー、認証/認可拒否、システムエラーが適切な応答・トランザクション結果・ログレベルで終了しているか。
- 検証可能性：正常系、業務エラー、権限拒否、入力異常、外部障害を自動テストまたは実環境で検証しているか。

### AI生成物レビュー観点

| 観点 | 確認内容 |
| --- | --- |
| 仕様外判断 | AIが仕様にない期待値、画面挙動、DB条件、権限制御を追加していないか |
| 既存整合 | 既存の命名、構成、責務分担、共通部品と矛盾していないか |
| 過剰実装 | 不要な抽象化、過剰な汎用化、対象外機能の先行実装がないか |
| テスト実効性 | 正常系だけに偏らず、異常系、境界値、権限、状態遷移を確認しているか |
| セキュリティ | 機密情報、固定パス、権限チェックのフロント依存、IDOR、CSRF漏れがないか |
| 記録 | AI出力の採用/不採用、修正内容、保留理由が作業記録に残っているか |

### AIレビュー指摘の扱い

AIの指摘はそのまま欠陥とせず、仕様根拠、テスト再現性、コード上の明確な不整合、セキュリティ・権限・データ整合性のリスクを確認する。根拠のない改善提案や存在しない仕様に基づく指摘は、不採用または確認事項として記録する。

## ユーザー追加指摘の反映

1. 今回限りの個別事情か、再発し得る一般的な観点かを判定する。
2. 一般的な観点は、該当する規約またはチェック表へ追加する。
3. 規約化しない場合は、理由を作業記録または指摘一覧に残す。


## 適用するレビュー観点の優先度

| 優先度 | 適用内容 |
| --- | --- |
| P0 | 仕様整合、正本ケース、期待結果、トレーサビリティ、テスト不足、テスト構成・責務境界、最終ゲート順序を確認する。 |
| P1 | 認証・認可・CSRF・入力検証、配置・共通化、静的解析・CI、ログ・観測可能性を変更内容に応じて確認する。ログは本資料の「ロガーレビューの共通観点」と`セキュリティ規約.md`を主資料とする。 |
| P2 | 性能、競合、互換性、アクセシビリティ、移行・復旧などを明示要件または変更リスクがある場合に確認する。 |

P1/P2を適用しない場合も、理由または保留条件をレビュー記録へ残す。
