# 実装後レビュー表

## 目的

AIが実装した成果物を、人間が確認しやすい単位でレビューする。

## チェック表

| 分類 | チェック内容 | 結果 | 指摘ID |
| --- | --- | --- | --- |
| 仕様整合 | 設計書・受入条件と実装が一致しているか |  |  |
| 対象範囲 | 仕様外の実装や過剰実装がないか |  |  |
| 入力チェック | バックエンドを正として検証しているか |  |  |
| 業務ルール | 状態遷移、計算、DB整合性が仕様通りか |  |  |
| 認証認可 | 未ログイン、ロール、所有者チェックが適切か |  |  |
| CSRF/セッション | 変更系APIでCSRFが効き、ログイン/ログアウトが安全か |  |  |
| エラー処理 | HTTPステータス、エラーコード、画面表示が妥当か |  |  |
| DB | Oracle前提、制約、マスタ、トランザクションが妥当か |  |  |
| 共通化 | 既存共通部品を使い、重複実装が増えていないか |  |  |
| ファイル配置 | feature別・責務別に探しやすい配置か |  |  |
| 命名 | Entity/DTO/Repository/Service/logicの名前が役割を表しているか |  |  |
| コメント | ファイル役割、分岐、難しい処理の意図コメントがあるか |  |  |
| 可読性 | 1ファイルに責務が詰まりすぎていないか、小さすぎる分割がないか |  |  |
| テスト | 正常系、異常系、境界値、権限、状態遷移が含まれるか |  |  |
| E2E | 主要な画面操作フローが確認されているか |  |  |
| カバレッジ | 未通過行に意味のある不足がないか |  |  |
| 静的解析 | lint、型チェック、ビルド、静的解析が通るか |  |  |
| 記録 | 指摘一覧、作業記録、レビュー記録が更新されているか |  |  |
| ユーザ追加指摘 | ユーザから追加指摘を受けた場合、指摘一覧へ記載するか、規約・チェック表・レビュー観点へ反映するかを明示判定したか |  |  |

## 指摘記録形式

```text
指摘ID:
分類:
重要度:
対象:
内容:
根拠:
修正案:
対応状況:
対応内容:
保留理由:
再確認条件:
```
