# ディレクトリ構成ルール

## 目的

チーム作業でファイル配置のばらつきを減らし、変更対象を探しやすくする。
機能別の責務と共通基盤の責務を分け、同じ種類の実装が複数箇所に散らばることを防ぐ。

## 基本方針

- 機能別の責務と共通基盤の責務を分ける。
- チーム並列開発を前提に、別機能が同一のController、Service、Repository、テストファイルへ継続的に変更を集めない。
- 画面、API、業務ルール、DBアクセス、型定義、テスト補助を同じ場所に混在させない。
- 新規ファイルは、既存の近い責務の配置を確認してから追加する。
- `src` 直下や共通パッケージ直下に、機能専用の実装を直接増やさない。
- 迷った場合は、配置理由と代替候補を作業記録に残す。

## フロントエンド

| 配置 | 置くもの | 注意点 |
| --- | --- | --- |
| `src/app` | アプリ起動、ルーティング、画面全体の接続 | 機能固有の入力検証やAPI詳細を置かない |
| `src/auth` | ログイン、ログアウト、認証状態、認証API | セッション、CSRF、認証エラーの扱いを集約する |
| `src/dailyReport` | 日報登録、編集、提出、日報固有UI/ロジック | 共通化できるAPI clientやフォーム部品を抱え込まない |
| `src/shared` | API client、共通型、共通UI、共通validation、共通test helper | 業務機能に依存しないものだけ置く |
| `e2e/support` | E2Eの認証モック、静的配信、共通操作 | specごとに同じ準備処理を直書きしない |

## バックエンド

| 責務 | 置くもの | 注意点 |
| --- | --- | --- |
| Controller | HTTP入力、認証済みユーザー取得、レスポンス変換 | 業務計算やDB更新手順を持たない |
| Service | 業務ルール、状態遷移、トランザクション境界 | HTTPや画面都合に依存しすぎない |
| Repository | DBアクセス、SQL、永続化 | 認可判断や画面表示ロジックを置かない |
| DTO | API入出力、画面との契約 | Entityをそのまま外部公開しない |
| Entity | DBまたはドメイン状態 | 画面専用項目を混ぜない |
| Common/Shared | 例外応答、認証補助、テスト補助、共通設定 | 機能専用の都合を入れない |

### 並列開発しやすい分割

- 同じ業務集約でも、登録・更新・提出などの更新系と、検索・一覧・詳細などの参照系は、変更理由が違う場合はController、Service、Repository、テストを分ける。
- 例: `DailyReportCommandController` / `DailyReportCommandService` と `DailyReportSearchController` / `DailyReportSearchService` のように、ユースケース単位で分割する。
- 既存ファイルに追加する場合は、同じファイルを他機能も更新する見込みがあるかを確認し、衝突しやすい場合は新規ファイルへ分離する。
- Entity、値オブジェクト、共通業務ルール、共通例外処理など、変更理由が同じものは無理に機能別へ複製しない。
- テストは本体以上に衝突しやすいため、機能追加時は既存の巨大なControllerテストへ追記し続けず、ユースケース別テストクラスへの分割を優先する。

## 確認観点

- 新規ファイルの配置理由を説明できるか。
- 同じ責務の実装が既存フォルダにないか。
- 別機能の開発者が同じファイルを更新し続ける構成になっていないか。
- 共通部品に機能専用の都合を混ぜていないか。
- 機能フォルダに共通化すべき処理を抱え込んでいないか。
- テスト補助やE2E supportがspecごとに重複していないか。
