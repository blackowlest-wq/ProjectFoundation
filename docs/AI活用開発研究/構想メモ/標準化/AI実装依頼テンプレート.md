# AI実装依頼テンプレート

## 目的

利用者が機能の目的、仕様、受入条件、制約を入力すると、AIが実装前確認からテスト、実装、レビュー、記録までを順番に実行するための入口を定義する。

詳細な実行内容は、次のAI内部資料を参照する。

- `AI内部プロンプト定義.md`: 実装前確認、シナリオ設計、テスト設計、単体テスト生成、実装、レビュー、記録
- `AI専門レビュー用プロンプト定義.md`: 観点別の専門レビュー

## 利用者向け依頼文

```text
以下の機能を実装してください。

対象機能:
{機能名・画面名・API名}

目的:
{利用者または業務上の目的}

仕様・受入条件:
{設計書、受入条件、画面/API仕様、業務ルール}

制約:
{既存構成、DB、認証認可、セキュリティ、性能、期限など}

対象外:
{今回変更しない範囲}

希望する確認:
{単体、API、画面、E2E、Oracle/DB、その他}
```

## AIの実行順

AIは、依頼を受けた後、次の順で実行する。

1. 実装前確認・仕様の曖昧点を整理する
2. BDD/E2Eシナリオを設計する
3. テスト観点と詳細テストケースを設計する
4. 承認済みケースから単体テストを生成する
5. 承認済みの仕様・ケースに基づいて実装する
6. セルフレビュー、専門レビュー、テスト実行、作業記録更新を行う

## 停止・承認条件

- 仕様、受入条件、対象範囲に曖昧さがある場合は、実装せず確認事項として停止する。
- シナリオと詳細テストケースを提示し、利用者の承認を得てからテストコード生成と実装へ進む。
- テストケースにない期待値や仕様をAIが推測する場合は、実装せず確認事項として停止する。
- 実装後は、テスト結果、未実行理由、レビュー指摘、保留事項を提示してから完了とする。
- 認証、認可、CSRF、DB、状態遷移など高リスク変更では、専門レビュー完了まで完了扱いにしない。

## 利用者への報告

各段階で、次の内容を簡潔に報告する。

- 実施した段階
- 作成または変更した資料・コード
- 受入条件、テストケース、実テストの対応
- 未確定事項、未実行テスト、保留理由
- 次の段階へ進むために必要な承認
