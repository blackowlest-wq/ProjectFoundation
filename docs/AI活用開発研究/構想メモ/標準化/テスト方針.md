# テスト方針

## 目的

チーム作業でテスト範囲と品質ゲートの判断を揃え、未確認のまま実装を進めることを防ぐ。

## 基本方針

- 業務ルール、入力検証、状態遷移、認証認可は自動テストを優先する。
- 画面操作の主要フローはE2Eで確認する。
- Mock E2Eは画面の操作・表示・分岐を安定して確認し、実Backend/Oracle E2EはログインからDB永続化・再読込までの接続実効性を確認する。両者は代替ではなく補完関係とする。
- 実Backend/Oracle E2Eでは、Backendログ、Frontendログ、Playwrightレポート、Oracle検証結果を取得し、相関ID・HTTP結果・業務状態変更を追跡できるようにする。認証情報を含み得るOracle E2EのPlaywright trace/video/screenshotは無効化し、Backendログを主な障害追跡記録とする。
- Oracle固有のDDL、SQL、制約、接続設定に関わる変更はOracle実機確認を検討する。
- テスト未実行の場合は、未実行理由と再確認条件を作業記録に残す。
- テストコードはWhat（何を検証し、何を期待するか）をテスト名、`describe`/`it`、Arrange/Act/Assertの構造で示す。実装手順の逐語説明は避けるが、テストデータの意図や複雑な前提がないとWhatを誤読する場合はコメントを残す。
- Oracle接続情報は `backend/config/oracle-test.properties` で管理し、バックエンドテストは共通のOracle入口を使用する。具体的な実行方法は [品質ゲート運用.md](品質ゲート運用.md) と `scripts/check.ps1` を参照する。

## テスト分類

| 分類 | 主な対象 | 目的 |
| --- | --- | --- |
| バックエンド単体/結合寄り | Controller、Service、Repository | 認証、入力検証、業務ルール、DB更新を確認する |
| フロント単体 | API client、validation、UI部品 | 画面ロジックと表示分岐を確認する |
| E2E | ログイン、日報登録、編集、提出など | ユーザー操作として主要フローが通ることを確認する |
| Mock E2E | APIを制御したFrontend操作 | UIの表示、入力、状態遷移を安定して確認する |
| Backend/Oracle E2E | 実Backend、実Oracle、実Frontend | 接続、認証、CSRF、DB保存、再読込、ログ相関を確認する |
| Oracle実機確認 | DDL、SQL、マスタ、制約 | 開発用DBとOracle差異による不具合を検出する |
| 静的解析 | TypeScript、Java、依存関係、secret | 型不整合、危険な実装、機密情報混入を検出する |

## テスト配置とユースケース分割

- フロント単体テストは `frontend/test` に配置する。
- フロントE2Eのspecとhelperは `frontend/e2e`、`frontend/e2e/support` に配置する。
- バックエンドの単体テスト・結合寄りテスト・`*IT.java` は `backend/src/test` に配置する。
- テストクラスは、本番のController / Service と同じユースケース境界に合わせ、検索/詳細、登録/更新、提出/再提出、マスタAPIなどの責務で分ける。
- 共通のログイン準備、fixture、seed投入は test support に置き、代表テストから必ず利用される形にする。
- `scripts/check-test-layout.ps1` は配置構造の品質ゲートであり、単体、結合寄り、E2E、typecheck、build、Oracle確認を置き換えるものではない。

## 品質ゲート

- 通常テストが成功している。バックエンドテストは共通のOracle入口から実行し、共通化されたOracle接続設定を読み込む。具体的な実行方法は [品質ゲート運用.md](品質ゲート運用.md) と `scripts/check.ps1` を参照する。
- TypeScript型チェックとフロントビルドが成功している。
- 変更対象の主要E2Eが成功している。
- 静的解析を導入済みの場合、重大以上の指摘が残っていない。
- Frontend / Backendのcoverageレポート確認は必須とし、Statements / Branches / Functions / Lines、およびINSTRUCTION / BRANCH / METHOD / LINEの4指標をそれぞれ85%以上に保つ。
- Oracle不要のBackend単体テストはPRのGitHub-hosted runnerで実行し、Oracle接続なしで業務ルール・共通例外・観測性・repository計算の回帰を検出する。
- Branches / BRANCHは分岐未充足を直接示すblocking指標として扱い、85%未満なら品質ゲート未通過とする。
- uncovered behaviorが見つかった場合は、その条件を再現するテストを追加し、同じcoverage入口を再実行して結果を更新する。
- 未実行の確認がある場合は、理由と再実行条件を作業記録に残す。

## テストコードのWhat確認

- Javaの`@Test`メソッド名から、対象操作、条件、期待結果が読める。
- Vitest/Playwrightの`describe`、`it`、`test`から、検証対象と期待結果が読める。
- テスト名で確認内容を表現できる場合、実装手順を説明するコメントを追加しない。
- テスト名を変更する場合も、期待値、アサーション、fixture、モック応答は変更しない。

## テスト追加の判断

| 変更内容 | 優先するテスト |
| --- | --- |
| 入力チェック変更 | バックエンドテスト、フロント単体テスト |
| 業務ルール変更 | Serviceテスト、Controllerテスト |
| 画面操作変更 | フロント単体テスト、E2E |
| 認証認可変更 | Controllerテスト、E2E |
| DB/SQL変更 | Repositoryテスト、Oracle実機確認 |
| 共通部品変更 | 利用元の代表テスト、回帰テスト |
