# テスト方針

## 目的

チーム作業でテスト範囲と品質ゲートの判断を揃え、未確認のまま実装を進めることを防ぐ。

## 基本方針

- 業務ルール、入力検証、状態遷移、認証認可は自動テストを優先する。
- 画面操作の主要フローはE2Eで確認する。
- Oracle固有のDDL、SQL、制約、接続設定に関わる変更はOracle実機確認を検討する。
- テスト未実行の場合は、未実行理由と再確認条件を作業記録に残す。

## テスト分類

| 分類 | 主な対象 | 目的 |
| --- | --- | --- |
| バックエンド単体/結合寄り | Controller、Service、Repository | 認証、入力検証、業務ルール、DB更新を確認する |
| フロント単体 | API client、validation、UI部品 | 画面ロジックと表示分岐を確認する |
| E2E | ログイン、日報登録、編集、提出など | ユーザー操作として主要フローが通ることを確認する |
| Oracle実機確認 | DDL、SQL、マスタ、制約 | 開発用DBとOracle差異による不具合を検出する |
| 静的解析 | TypeScript、Java、依存関係、secret | 型不整合、危険な実装、機密情報混入を検出する |

## テスト配置とユースケース分割

- フロント単体テストは `frontend/test` に配置する。
- フロントE2Eのspecとhelperは `frontend/e2e`、`frontend/e2e/support` に配置する。
- バックエンドの単体テスト・結合寄りテスト・`*IT.java` は `backend/src/test` に配置する。
- テストクラスは、本番のController / Service と同じユースケース境界に合わせ、検索/詳細、登録/更新、提出/再提出、マスタAPIなどの責務で分ける。
- 共通のログイン準備、fixture、seed投入は test support に置き、代表テストから必ず利用される形にする。
- `scripts/check-test-layout.ps1` は配置構造の品質ゲートであり、単体、結合寄り、E2E、typecheck、build、Oracle確認を置き換えるものではない。

## 品質ゲート

- 通常テストが成功している。
- TypeScript型チェックとフロントビルドが成功している。
- 変更対象の主要E2Eが成功している。
- 静的解析を導入済みの場合、重大以上の指摘が残っていない。
- カバレッジが大きく悪化した場合は理由を確認する。
- 未実行の確認がある場合は、理由と再実行条件を作業記録に残す。

## テスト追加の判断

| 変更内容 | 優先するテスト |
| --- | --- |
| 入力チェック変更 | バックエンドテスト、フロント単体テスト |
| 業務ルール変更 | Serviceテスト、Controllerテスト |
| 画面操作変更 | フロント単体テスト、E2E |
| 認証認可変更 | Controllerテスト、E2E |
| DB/SQL変更 | Repositoryテスト、Oracle実機確認 |
| 共通部品変更 | 利用元の代表テスト、回帰テスト |
