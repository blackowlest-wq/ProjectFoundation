# 実装後レビュー観点

## 目的

レビュー担当者による確認観点のばらつきを減らし、仕様、セキュリティ、保守性、テスト不足を安定して検出する。

## 基本方針

- 差分の見た目だけでなく、変更理由と影響範囲を確認する。
- 仕様、責務、セキュリティ、テスト、記録の順に確認する。
- 問題を検出した場合は、作業記録だけでなく指摘一覧にも残す。
- 修正しない場合は、保留理由と再確認条件を明記する。
- ユーザから追加指摘を受けた場合は、再発防止の観点として規約やレビュー観点へ追加すべきか確認する。
- 本体コードはHow、テストコードはWhat、コミットログはWhy、コードコメントは必要なHowとWhy notという役割分担で確認する。
- コメントの有無ではなく、ファイル責務・境界の説明が必要か、分岐・重要処理のHowがコードだけで誤読されないか、Why notコメントの根拠があるかを確認する。
- 各メソッド・関数は、直前の日本語コメントから責務・入力・出力・副作用・例外・分岐の契約が分かるかを確認する。単純なアクセサや一行委譲は、コメント省略の妥当性を確認する。
- `if`/`else`の各分岐は、処理手順、優先順位、早期return、状態変更、許可・拒否の結果を示す`How:`があるか確認する。`Why not:`だけの場合は、方式の除外理由と分岐手順を混同した指摘として扱う。

## レビュー観点

| 観点 | 確認内容 |
| --- | --- |
| 仕様整合 | 設計書、受入条件、作業依頼と実装が一致しているか |
| 対象範囲 | 仕様外の実装や過剰実装がないか |
| 責務分離 | Controller、Service、Repository、UI、API clientの責務が混ざっていないか |
| 認証認可 | 未ログイン、ロール、所有者、他者データアクセスが制御されているか |
| CSRF/セッション | 変更系APIでCSRFが有効か、ログイン/ログアウト後の状態が安全か |
| 入力検証 | フロントだけでなくバックエンドを正として検証しているか |
| DB整合性 | 制約、トランザクション、マスタ参照、Oracle前提が崩れていないか |
| エラー処理 | ユーザー表示、HTTPステータス、ログ、例外分類が適切か |
| ログ | 機密情報を出していないか、調査に必要な情報は残るか |
| 共通化 | 既存共通部品を使い、重複実装が増えていないか |
| 可読性 | 1ファイルに責務が詰まりすぎていないか、小さすぎる分割がないか |
| 成果物の役割 | 本体コードのHow、テストコードのWhat、コミットログのWhyがそれぞれの成果物から読めるか |
| コメント | 必要な責務・境界の冒頭コメントと分岐・重要処理の`How:`があり、冗長なHowや根拠のない`Why not:`がないか |
| メソッド・関数コメント | 公開API、状態変更、DBアクセス、外部通信、例外・複数分岐を持つメソッド・関数の責務と契約が直前の日本語コメントで分かるか |
| if分岐コメント | `if`/`else`の処理手順、優先順位、早期return、状態変更、許可・拒否の結果が`How:`から分かるか。`Why not:`だけになっていないか |
| テスト表現 | テスト名・`describe`/`it`・アサーションからWhatと期待結果が読めるか |
| コミットログ | コミット本文に背景・目的・影響のWhyがあるか |
| 並列開発性 | 別機能が同じController、Service、Repository、テストファイルを同時更新しやすい構成になっていないか。ユースケース単位で分割すべき箇所がないか |
| テスト混在 | `frontend/src` / `backend/src/main` にテストが混在していないか |
| ユースケース再集中 | 一つのController、Service、テストクラスへ複数ユースケースを再集中させていないか |
| テスト移動差分 | テストの移動・分割で確認内容、期待値、アサーション、SQL fixture が意図せず変わっていないか |
| 配置チェック実行 | `scripts/check-test-layout.ps1` を実行し、違反がないことを確認したか |
| テスト | 正常系、異常系、境界値、権限、状態遷移が確認されているか |
| 記録 | 作業記録、指摘一覧、レビュー記録が必要に応じて更新されているか |

## AI生成物レビュー観点

| 観点 | 確認内容 |
| --- | --- |
| 仕様外判断 | AIが仕様にない期待値、画面挙動、DB条件、権限制御を追加していないか |
| 既存整合 | 既存の命名、構成、責務分担、共通部品と矛盾していないか |
| 過剰実装 | 不要な抽象化、過剰な汎用化、対象外機能の先行実装がないか |
| テスト実効性 | 正常系だけに偏らず、異常系、境界値、権限、状態遷移を確認しているか |
| セキュリティ | 機密情報、固定パス、権限チェックのフロント依存、IDOR、CSRF漏れがないか |
| 記録 | AI出力の採用/不採用、修正内容、保留理由が作業記録に残っているか |

## AIレビュー指摘の扱い

AIのレビュー指摘は、そのまま欠陥として扱わない。
次のいずれかを満たす場合に有効な指摘候補として扱う。

- 仕様根拠が確認できる。
- テストで再現できる。
- コード上の明確な不整合がある。
- セキュリティ、権限、データ整合性のリスクが説明できる。

次の指摘は、不採用または確認事項として扱う。

- 仕様根拠がない改善提案。
- プロジェクト方針と合わない一般論。
- 存在しないコードや仕様に基づく指摘。
- 実装済み内容を見落とした指摘。
## 指摘記録

指摘は次の情報を残す。

```text
指摘ID:
分類:
重要度:
対象:
内容:
根拠:
修正案:
対応状況:
対応内容:
保留理由:
再確認条件:
```

## ユーザ追加指摘の反映

ユーザから追加指摘を受けた場合は、次の順で確認する。

1. 指摘が今回限りの個別事情か、今後も再発し得る一般的な観点かを判断する。
2. 一般的な観点の場合は、該当する規約やチェック表に追加する。
3. 規約化する場合は、具体的な確認観点として書き、個別事例だけをそのまま残さない。
4. 規約化しない場合は、理由を作業記録または指摘一覧に残す。

反映先の目安は次の通り。

| 指摘の種類 | 主な反映先 |
| --- | --- |
| ファイル配置、責務分離、並列開発時のファイル衝突 | `ディレクトリ構成ルール.md` |
| レビュー漏れ、確認観点不足 | `実装後レビュー観点.md` |
| 実装前の確認不足 | `実装前確認観点.md` |
| テスト不足、確認コマンド不足 | `テスト方針.md` |
| 認証、認可、CSRF、入力検証、ログ | `セキュリティ規約.md` |
| 重複実装、共通化不足、過剰共通化 | `共通部品化判断基準.md` |
