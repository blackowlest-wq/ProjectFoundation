# 品質ゲート運用

## 正本の役割

| 資料・入口 | 役割 |
| --- | --- |
| `テスト方針.md` | 何を、なぜテストするか |
| `テスト・静的解析チェック表.md` | 観点とツールの対応 |
| `品質ゲート運用.md` | いつ、どのMode・CI jobを使うか |
| `scripts/check.ps1` | 実コマンド、実行順、終了判定 |

具体的なコマンドはこの資料と`check.ps1`に集約し、他の資料へ重複して記載しない。

## 固定方針

- rootと`frontend`は独立npmプロジェクトであり、workspaces化しない。
- Node.js 24.18.0 / npm 11.16.0、Java 21、Maven core 3.9.16、Maven Wrapper 3.3.4を固定する。
- Lefthook 2.1.10、Gitleaks 8.30.1、Playwright 1.58.0も固定する。
- Git hookは早期フィードバック、最終判定はGitHub Actionsの品質ゲートとする。
- workflowは`permissions: contents: read`、Actionは完全SHA固定とする。

## モジュール追加時の不変契約

モジュール追加時に、`scripts/check.ps1`やGitHub Actions workflowへ個別moduleのパス・名前を追記しない。品質ゲートは、集約ビルドとテスト命名規約に委譲する。

- Backendの複数module化では、`backend/pom.xml`をMaven aggregatorとし、`check.ps1`はaggregatorの標準ライフサイクルと静的解析を呼ぶ。新moduleのPOMをrunnerやworkflowへ追加しない。
- 現在のFrontendはrootと`frontend`の独立npmプロジェクトである。複数packageを導入する段階でnpm workspaceとrootのaggregate scriptへ移行し、その設計変更時だけ`check.ps1`の呼び出し境界を更新する。以後はworkspace rootを呼び、package単位の列挙をしない。
- Javaテストは通常`*Test` / `*Tests` / `*TestCase`、Oracle等の統合テストは`*IT`、Frontend単体テストは`*.test.*` / `*.spec.*`、E2EはPlaywrightの設定済みtestDirへ置く。追加moduleはこの規約に従うことで既存aggregate testに含める。
- 必須のlint、test、buildがmodule側で未定義の場合、aggregate scriptまたはmodule定義を失敗させる。任意処理だけを`--if-present`とし、未実行を成功扱いにしない。
- 集約方式やテスト発見規約自体を変更する場合は、module追加とは別の設計変更としてrunner、workflow、CI必須statusの影響を一度レビューする。

## Local Mode

| Mode / task | 用途 | Oracle接続 |
| --- | --- | --- |
| `Quick` | staged対象の空白、生成物、secret、Frontend lint、Markdown lint、Java変更時Spotless | なし |
| `Full` | Frontend lint / typecheck / unit / build、Backend test-compile / Spotless / Checkstyle / SpotBugs、coverage gate contract | なし |
| `Oracle` | 接続先・環境・ユーザーの前段ガード、安全識別テスト、通常テストと`*IT` | あり |
| `All` | `Full`完了後に`Oracle`を実行し、両方の失敗を集約 | あり |
| `FrontendCoverage` | Frontend Vitest coverage | なし |
| `BackendCoverage` | JaCoCo付きBackend test（Oracle接続） | あり |
| `E2E` | production build後のPlaywright E2E | なし |
| `E2EOracle` | Backend + Oracle + Vite previewを起動した実接続Playwright E2E | あり |
| `DirectorySecrets` | 依存取得・build前のdirectory Gitleaks | なし |
| `DependencyAudit` | root/frontend npm auditとOWASP Dependency-Check | なし。ただしNVD更新が必要 |

実行入口は次のとおりとする。

```powershell
pwsh -NoProfile -File scripts/doctor.ps1
pwsh -NoProfile -File scripts/check.ps1 -Mode Quick
pwsh -NoProfile -File scripts/check.ps1 -Mode Full
pwsh -NoProfile -File scripts/check.ps1 -Mode Oracle
pwsh -NoProfile -File scripts/check.ps1 -Mode All
pwsh -NoProfile -File scripts/check.ps1 -CiTask FrontendCoverage
pwsh -NoProfile -File scripts/check.ps1 -CiTask BackendCoverage
pwsh -NoProfile -File scripts/check.ps1 -CiTask E2E
pwsh -NoProfile -File scripts/check.ps1 -CiTask E2EOracle
pwsh -NoProfile -File scripts/check.ps1 -CiTask DirectorySecrets
pwsh -NoProfile -File scripts/check.ps1 -CiTask DependencyAudit
```

`Oracle` / `BackendCoverage`は、Git管理外の`backend/config/oracle-test.properties`または
self-hosted runnerの保護されたprocess環境変数を使う。資格情報をコマンドライン、ログ、Gitへ出さない。

- `FrontendCoverage`はテスト実行後に`frontend/coverage/index.html`、`coverage-summary.json`、`lcov.info`の存在を確認する。テスト失敗またはthreshold未達は非0終了とし、レポート確認は同じtask内で続けて実行する。
- `BackendCoverage`はOracle wrapper経由でMaven `verify` を実行し、その後に`backend/target/site/jacoco/index.html`、`jacoco.xml`、`jacoco.csv`の存在を確認する。Oracle接続情報が不足した場合も、失敗を成功扱いにせず未実行理由と再確認条件を記録する。
- CIのcoverage artifactは失敗時も回収できるように`if: always()`を維持し、Frontendは`frontend/coverage`、Backendは`backend/target/site/jacoco`をupload対象に固定する。

## Quickの対象選択

QuickはGit indexからstagedファイル名を取得する。ESLint、Markdownlint、Java静的解析へ渡す対象名は
indexから選ぶが、検査対象の内容は作業ツリーから読む。したがって、staged後に作業ツリーだけ変更した
内容も検査される。自動修正、自動stage、`stage_fixed`は行わない。

## OracleとDDLの安全条件

- `DAILY_REPORT_DB_ENV=TEST`、想定host / service / DB name / userを必須とする。
- 接続ユーザーは想定テストユーザーと一致し、`SYS`、`SYSTEM`、本番相当の名前は拒否する。
- JDBC URLは想定host / serviceと照合し、パスワードと完全URLは出力しない。
- 接続後に`SYS_CONTEXT`でDB名、service、session userを確認する。
- Oracle識別子はDBが返す大小文字表現を正規化して比較するが、host / service / DB名 / session userの
  完全な論理値が一致することを必須とする。部分一致や曖昧な前方一致は許可しない。
- アプリ初期化DDLは`DAILY_REPORT_ALLOW_DDL=true`が必要で、任意DDLスクリプトはさらに
  `-AllowDdl`と許可ディレクトリ内の実在`.sql`指定を必要とする。
- Oracle workflowは`push main`と手動実行だけとし、`pull_request`では起動しない。
- self-hosted runnerは個人環境から隔離した専用VMまたは端末、非管理者サービスアカウント、専用test schemaとする。
- self-hosted runner上でPRコードを直接実行しない。PRのFullはGitHub-hosted runnerで実行し、Oracleはmainへ反映後に実行する。

## CI jobとbranch protection

| workflow / job | check-run名 | 実行環境 | 役割 |
| --- | --- | --- | --- |
| `quality.yml / full-windows` | `Full / Windows` | GitHub-hosted Windows | Windows Full |
| `quality.yml / full-linux` | `Full / Linux` | GitHub-hosted Linux | Linux Full |
| `quality.yml / backend-unit` | `Backend / Unit` | GitHub-hosted Linux | Oracle-independent Backend unit tests |
| `quality.yml / coverage-frontend` | `Coverage / Frontend` | GitHub-hosted Linux | Frontend coverage |
| `quality.yml / e2e` | `E2E` | GitHub-hosted Linux | Playwright E2E |
| `quality.yml / gitleaks-directory` | `Gitleaks / Directory` | GitHub-hosted Linux | directory secret scan |
| `security.yml / history-gitleaks` | — | GitHub-hosted Linux | 週次Git履歴scan |
| `security.yml / dependency-audit` | — | GitHub-hosted Linux | 週次依存監査 |
| `oracle.yml / oracle-integration` | — | 隔離self-hosted | Oracle safety + 64件相当のintegration |
| `oracle.yml / oracle-coverage` | — | 隔離self-hosted | Oracle-backed Backend coverage |
| `oracle.yml / oracle-e2e` | — | 隔離self-hosted | 実Backend + Oracle + FrontendのPlaywright E2E、Backendログ・Oracle検証・レポート artifact |

Oracleの3ジョブは、テスト開始前に`doctor-backend-oracle.ps1`を実行する。事前診断は、保護された接続情報の存在、`DAILY_REPORT_DB_ENV=TEST`、Oracle設定例、Java 21、Maven 3.9.16を確認し、接続値やパスワードを出力しない。`oracle-coverage`はJavaセットアップとMaven依存準備を明示的に行い、3ジョブとも実行時間を30分に制限する。

CODEOWNERSは所有者候補を示すだけで、レビュー必須化はしない。GitHub Settingsのmain branch protectionで、
次を別途設定する。

- Require a pull request before merging
- Require review from Code Owners
- Dismiss stale pull request approvals when new commits are pushed
- Require status checks: `Full / Windows`、`Full / Linux`、`Backend / Unit`、`Coverage / Frontend`、`E2E`、`Gitleaks / Directory`
- mainへのforce pushを禁止
- mainのbranch deletionを禁止

2026-07-18時点で、public repositoryの`main`へ上記6 check、PR必須、Code Ownerレビュー1件、stale approval破棄、administrator enforcementを設定済みである。

Oracle statusを必須化する場合は、Oracle環境が常時利用できることと、main push/manual運用との整合を確認してから追加する。

## 複数実装の検証

同じ設計書から複数実装を作る場合は、設計書内へ`v1`、`v2`のソースを同居させず、baseline commitから実験ごとのbranchまたはworktreeを作る。
生成物、coverage、Oracle実設定はGit管理外にし、各branchでQuick / Full / Oracleの結果とcommitを作業記録へ残す。

## 未実行時の再確認条件

- PowerShell、Node/npm、Java、Maven、Lefthook、Gitleaks、Playwrightの固定版が揃わない場合はdoctorを先に再実行する。
- Oracleは専用runner、保護環境、expected identity、test schema、Java 21、Maven 3.9.16が揃ったmain pushまたは手動実行で再確認する。事前診断が失敗した場合はOracleへ接続せず、不足した環境変数名または前提条件だけを確認して補正する。
- Dependency-CheckはNVD更新が完了するまで成功扱いにしない。初回更新が長時間化した場合はキャッシュ済み環境または更新時間を確保したCIで再実行する。
- 実GitHub Actions jobは、workflowをpushし、branch protectionとself-hosted runner label / environment secretを設定した後に再確認する。
