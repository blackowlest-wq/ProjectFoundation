# 開発フロー

## 目的

AIを使った機能開発で、実装前の確認不足、テスト不足、レビュー指摘の記録漏れ、過剰なファイル分割や共通化漏れを減らす。

この資料は、ProjectFoundationにおけるAI活用開発の正本フローである。

## 基本原則

- PlanなしでDoしない。
- 設計確認なしで実装しない。
- 作業内容に応じて標準化資料を確認し、確認できなかった場合は理由を記録する。
- AIに設計判断を丸投げしない。
- AIに渡す前に、入力情報を整理する。
- AI生成物は人間レビューなしで採用しない。
- AIが仕様外の判断をした場合は、実装せず確認事項として扱う。
- 受入条件からBDDシナリオを作る。
- 業務ルール、入力チェック、状態遷移、計算ロジックはTDD寄りに先にテストを考える。
- 機能実装と大きなリファクタを同じ変更単位に混ぜすぎない。
- レビュー指摘は作業記録だけでなく指摘一覧へ記録する。
- ユーザから追加指摘を受けた場合は、指摘一覧へ記載するか、規約・チェック表・レビュー観点へ反映するかをその場で明示判定する。
- 生成物、カバレッジ、テスト結果をソース実装と混同しない。
- AI利用に伴う確認・修正時間も作業実績に含める。
- 成果物ごとに記載する情報を分ける。本体コードはHow、テストコードはWhat、コミットログはWhy、コードコメントは必要なHowとWhy notを担当する。
- 本体コードの処理手順は、構造・型・命名で表現できる範囲を明確にする。ただし、分岐の優先順位、アルゴリズム、処理順、重要な不変条件、副作用、外部APIとの接続など、コードだけでは誤読しやすいHowはコメントで残す。
- 代替方式を採用しなかった理由、制約、互換性、セキュリティ上の除外理由は`Why not:`で残す。
- ファイル冒頭コメントは、ファイル名だけでは分からない責務、境界、公開契約を示す場合に残す。クラス名やフィールドの単純な言い換えは書かない。
- 各メソッド・関数は、直前の日本語コメントで責務または公開契約を示す。特に公開API、状態変更、DBアクセス、外部通信、例外・早期return、複数分岐を持つ処理は、入力・出力・副作用・分岐の意味まで記載する。
- `if`/`else`の分岐は、条件に応じた処理手順、優先順位、早期return、状態変更、許可・拒否の結果を示す`How:`を分岐の直前または分岐内に残す。`Why not:`だけでは分岐の実行手順を説明したことにしない。
- 単純なコンストラクタ、getter/setter、recordアクセサ、名前だけで責務と契約が明確な一行委譲は、コメントを省略してよい。省略時も、呼び出し側が誤読しないかをレビューする。

## 成果物ごとの記載ルール

| 成果物 | 記載する内容 | 記載しない内容 |
| --- | --- | --- |
| 本体コード | 構造、型、命名、実装そのものによるHow | コードを逐語的に説明するコメント |
| テストコード | テスト名、`describe`/`it`、Arrange/Act/AssertによるWhatと期待結果 | 実装手順を説明するコメント |
| コミットログ | 背景、目的、利用者・保守への影響などのWhy | 変更ファイルの列挙だけ、実装手順だけの本文 |
| コードコメント | 誤読しやすい処理のHow、採用しなかった方式と除外理由（Why not）、必要な責務・契約 | コードを逐語的に説明するHow、What、仕様の単純な転記 |
| メソッド・関数コメント | 責務、入力・出力、状態変更、DB/外部通信、例外・分岐の契約 | メソッド本体を上から言い換えるだけの説明 |
| if分岐コメント | 条件に応じた処理手順、優先順位、早期return、状態変更、許可・拒否の結果を示す`How:` | `Why not:`だけで分岐手順を説明したつもりにすること |

コードコメントを残す場合は、次の形式を基準にする。

```text
How: [分岐の優先順位・アルゴリズム・処理順・副作用など、コードだけでは誤読しやすい実装上の要点]。
Why not: [採用しなかった方式]を選ばず、[制約・互換性・セキュリティ・保守上の理由]により現在の方式を採用する。
```

ESLintなどのツール指示コメントは維持してよいが、直前に抑制理由を`Why not:`で記載する。既存コメントを削除または書き換えた場合は、作業記録へ判断理由を残す。

## 標準フロー

```text
1. Intake
   ユーザー依頼、対象機能、対象外、制約、既存変更を確認する。

2. Plan
   `標準化資料一覧.md`を入口に、`実装前チェック表.md`、`テスト方針.md`、対象機能の設計書・受入条件・DB/API資料、作業記録を読み、実装計画を作る。

3. Design Check
   `実装前チェック表.md`を基準に、仕様、受入条件、入力チェック、状態遷移、非機能、セキュリティ、DB整合性を照合する。認証・認可・CSRF・入力を扱う場合は`セキュリティ規約.md`、ログを扱う場合は`実装前チェック表.md`の「ログ設計」と`セキュリティ規約.md`のログ節を主資料とする。配置・共通化を扱う場合は`ディレクトリ構成ルール.md`と`共通部品化判断基準.md`を追加で読む。

4. BDD / Test Design
   `テストケースレビュー観点.md`、`テスト方針.md`、`サンプル設計書/テスト設計書.md`を基準に、受入条件からBDDシナリオと正本テストケースを作る。ケースレビューで仕様根拠、期待結果、テスト層、テスト構成・責務境界、対象外・保留を確認し、テスト構成・責務対応表を作成してから、単体、結合寄り、E2E、静的解析の確認範囲を決める。

5. Do
   小さく実装する。`ディレクトリ構成ルール.md`、`セキュリティ規約.md`、`共通部品化判断基準.md`に合わせる。作業記録は統合担当が`作業記録テンプレート.md`に沿って更新する。

6. Self Review
   `実装後レビュー表.md`に沿って、仕様、テスト、テスト構成・責務分割、セキュリティ、可読性、ファイル配置、共通化、How/What/Why/Why not、記録を自己確認する。Full実行前に構成判定を終える。

7. Independent Review
   機能追加や大きめの修正では、`AI専門レビュー用プロンプト定義.md`の「01 設計照合レビュー」「02 テスト不足レビュー」「06 期待結果レビュー」「08 トレーサビリティ・統合レビュー」をP0として独立にレビューする。認証・認可・CSRF・入力検証は同「03 セキュリティレビュー」と`セキュリティ規約.md`、可読性・配置・共通化は同「04 可読性・配置レビュー」と`ディレクトリ構成ルール.md`/`共通部品化判断基準.md`、CIは同「05 静的解析・CIレビュー」と`テスト・静的解析チェック表.md`/`品質ゲート運用.md`をP1として変更時に追加する。ログ・観測可能性は`実装後レビュー表.md`の「ロガーレビューの共通観点」と`セキュリティ規約.md`のログ節を使う。P2の非機能観点は明示要件または変更リスクがある場合だけ追加する。2観点以上は並列実行し、最後に統合担当を置く。各担当へ渡すファイル、節、対象範囲、出力先は、`skills/projectfoundation-preflight-ja/SKILL.md`または`skills/projectfoundation-review-ja/SKILL.md`の「セッションをまたぐレビュー参照契約」に従って記録する。

8. Fix
   指摘を修正し、対応内容と保留理由を記録する。

9. Verify
   独立レビューと指摘修正を完了した後、テスト構成・責務分割の判定を確認し、テスト方針に沿って、バックエンド通常テスト、フロントテスト、型チェック、ビルド、E2E、カバレッジ、Oracle確認を最終ゲートとして実行する。レビュー前の実行はベースラインと記録する。

10. Record
    統合担当が`作業記録テンプレート.md`に沿って、作業記録、指摘一覧、今回専用のレビュー記録、正本テストケース、受入条件レビューを更新する。各指摘に標準化判定、反映先、保留理由、再確認条件を記録する。ユーザ追加指摘がある場合は、指摘一覧への記載要否と規約・チェック表・レビュー観点への反映要否を記録する。
```

## フェーズ別ゲート

| フェーズ | 次へ進む条件 |
| --- | --- |
| Intake | 依頼内容、対象機能、既存変更の有無が分かっている |
| Plan | 参照設計書、既存実装、DB/API、標準化資料、テスト方針、保留事項が整理されている |
| Design Check | 仕様根拠のある実装範囲と、確認が必要な曖昧点が分離されている |
| BDD / Test Design | 受入条件ごとの確認方法、正本ケースの期待結果、テスト構成・責務対応表がレビュー済みである |
| Do | 変更単位が小さく、既存構成と命名に合っている |
| Self Review | 実装後レビュー表に沿ったセルフレビューが終わっている |
| Independent Review | 必須観点が独立にレビューされ、統合結果と標準化判定が記録されている |
| Verify | 指摘修正後の最終ゲートが成功している、または未実行理由と再確認条件が記録されている |
| Record | 作業記録、指摘一覧、レビュー記録、正本ケース、受入条件レビューが揃い、標準化判定が明示されている |

## BDD / TDDの使い分け

| 対象 | 推奨 |
| --- | --- |
| 受入条件、業務フロー | BDDでシナリオ化する |
| 入力チェック、状態遷移、時刻計算、権限制御 | TDD寄りに先にテストケースを作る |
| 画面操作の主要フロー | E2Eで代表シナリオを確認する |
| リファクタ | 既存テストを先に通し、必要なら保護テストを追加する |

## 変更単位の分け方

- 機能追加、リファクタ、ファイル移動、コメント追加、テスト追加は可能な範囲で分ける。
- 同じ変更に含める場合は、作業記録で理由を明記する。
- 大きな移動や改名をした後は、機能変更を入れる前にテストを実行する。

## 独立レビュー利用方針

- サブエージェントは、必要に応じた観点別レビューや独立調査に限定して使う。
- サブエージェントには期待する答えを渡さず、対象資料と観点だけを渡す。
- 指摘は「仕様根拠あり」「テストで再現可能」「コード上明確」「一般論・保留」に分類する。

## 静的解析・品質ゲート

最低限:

- TypeScript型チェック
- フロントエンドテスト
- フロントエンドビルド
- バックエンド通常テスト
- E2E
- カバレッジ

導入候補:

- ESLint / typescript-eslint
- Prettier
- Checkstyle または Spotless
- SpotBugs
- JaCoCo threshold
- Vitest coverage threshold
- CodeQL または同等のcode scanning
- secret scanning
- dependency check

```text

AIへの依頼内容:
AI出力の採用有無:
採用した内容:
修正した内容:
不採用理由:
確認者:
```
