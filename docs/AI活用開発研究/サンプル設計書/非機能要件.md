# サンプルアプリ 非機能要件

## 目的

社内向け日報管理システムのサンプル開発における、システム固有の非機能要件を定義する。

本資料は、Java / Spring Boot バックエンド、React + TypeScript フロントエンド、Oracle Databaseを利用した実装、テスト、AIレビューの入力情報として使用する。
関連資料は以下を参照する。

- 機能一覧・受入条件: `機能一覧・受入条件.md`
- 画面設計: `画面設計.md`
- API一覧: `API一覧.md`
- DB概念設計: `DB概念設計.md`
- 状態遷移・業務フロー設計: `状態遷移・業務フロー設計.md`

- テスト設計書: `テスト設計書.md`

## 前提

- 初回サンプルは、AI活用開発の研究・検討を目的とする社内向けWebアプリとする
- 外部公開インターネットサービスではなく、社内利用を想定する
- 個人情報として、社員ID、社員名、勤務実績、案件別作業時間を扱う
- パスワードは平文保持しない
- DBはOracle Databaseを使用する
- 本資料は初回サンプルで満たす最低限の非機能要件を定義し、本格導入時の全社標準を代替しない

## 非機能要件一覧

| 分類 | 要件ID | 要件 | 優先度 |
| --- | --- | --- | --- |
| セキュリティ | NF-SEC-001 | 認証済み利用者のみ業務画面と業務APIを利用できる | 必須 |
| セキュリティ | NF-SEC-002 | 権限制御はフロントエンド表示制御だけでなく、バックエンドAPIで必ず実施する | 必須 |
| セキュリティ | NF-SEC-003 | パスワードはハッシュ化して保持し、ログやレスポンスに出力しない | 必須 |
| セキュリティ | NF-SEC-004 | 入力値はバックエンドで検証し、仕様外の値をDBに保存しない | 必須 |
| セキュリティ | NF-SEC-005 | XSS、CSRF、SQLインジェクションを考慮した実装にする | 必須 |
| 性能 | NF-PERF-001 | 主要画面の通常操作は、初回サンプルの想定データ量で実用上待たされない応答にする | 必須 |
| 性能 | NF-PERF-002 | 月次集計とCSV出力は、承認済み日報のみを対象にして不要な全件処理を避ける | 必須 |
| 可用性 | NF-AVL-001 | 初回サンプルでは冗長構成、バックアップ運用、災害対策は対象外とする | 対象外 |
| 保守性 | NF-MNT-001 | 仕様上の状態、ロールはEnumとして扱い、文字列の直書きを局所化する。休日区分、休憩区分、グループはマスタとして扱う | 必須 |
| 監査 | NF-AUD-001 | 提出、承認、差戻し、再提出に必要な監査項目を記録する | 必須 |
| ログ | NF-LOG-001 | エラー原因調査に必要なログを出力し、パスワードなどの機密情報は出力しない | 必須 |
| UX | NF-UX-001 | 入力エラー、権限エラー、状態不整合は利用者が理解できる形で表示する | 必須 |

## 認証・セッション

| 要件ID | 要件 |
| --- | --- |
| NF-AUTH-001 | ログイン成功後のみ、日報一覧、日報登録、日報詳細、未承認一覧、月次集計、CSV出力を利用できる |
| NF-AUTH-002 | 未ログイン状態で保護されたAPIを呼び出した場合は `401` を返す |
| NF-AUTH-003 | ログアウト後は、保護された画面とAPIを利用できない |
| NF-AUTH-004 | セッション方式を採用する場合、Cookieには `HttpOnly`、`Secure`、`SameSite` を設定する |
| NF-AUTH-005 | 初回サンプルではパスワード変更、パスワードリセット、アカウントロック、多要素認証は対象外とする |

## 認可

| 要件ID | 要件 |
| --- | --- |
| NF-AUTHZ-001 | 社員は自分の日報のみ参照、登録、編集、提出、再提出できる |
| NF-AUTHZ-002 | 上長は承認対象グループの日報のみ参照、承認、差戻しできる |
| NF-AUTHZ-003 | 管理者は全社員の日報参照、月次集計、CSV出力を利用できる |
| NF-AUTHZ-004 | 管理者は日報の登録、編集、提出、再提出、承認、差戻しを行わない |
| NF-AUTHZ-005 | 権限外の操作は `403` を返し、対象データを更新しない |
| NF-AUTHZ-006 | 一覧・検索APIでは、権限外データをレスポンスに含めない |

## 入力値検証

| 要件ID | 要件 |
| --- | --- |
| NF-VAL-001 | 入力値検証はバックエンドで必ず実施する |
| NF-VAL-002 | フロントエンドの入力制御は補助とし、バックエンド検証の代替にしない |
| NF-VAL-003 | 日付、年月、時刻、時間量、Enum値は `入力チェック・業務ルール一覧.md` に従って検証する |
| NF-VAL-004 | 入力エラー時は `400` を返し、DB更新を行わない |
| NF-VAL-005 | 状態不整合時は `409` を返し、DB更新を行わない |
| NF-VAL-006 | エラー時は、画面に表示できるエラーコード、メッセージ、項目別詳細を返す |
| NF-VAL-007 | 休憩時間はリクエスト値を信用せず、社員の休憩区分からバックエンドで算出する |

## Webセキュリティ

| 要件ID | 要件 |
| --- | --- |
| NF-WEB-001 | 画面に表示する備考、差戻しコメント、社員名、案件名、作業分類名はHTMLとして解釈せず、文字列として表示する |
| NF-WEB-002 | SQLはパラメータバインドまたはORMの安全なAPIを利用し、文字列連結で組み立てない |
| NF-WEB-003 | 状態変更APIはCSRF対策を行う。Cookieセッションを利用する場合はCSRFトークンを使用する |
| NF-WEB-004 | レスポンスにパスワードハッシュ、不要な内部ID、スタックトレースを含めない |
| NF-WEB-005 | CORSは必要最小限のオリジンに限定する。初回サンプルでは開発環境のみに限定する |

## エラー処理

| 要件ID | 要件 |
| --- | --- |
| NF-ERR-001 | APIエラーは `API一覧.md` の共通エラーレスポンス形式に従う |
| NF-ERR-002 | 画面では、入力エラー、権限エラー、状態不整合、対象データなしを区別して表示する |
| NF-ERR-003 | 想定外エラー時は、利用者に内部情報を表示しない |
| NF-ERR-004 | 想定外エラーはログに記録し、調査可能にする |
| NF-ERR-005 | 状態変更の途中でエラーが発生した場合、状態と監査項目の一部だけが更新された状態にしない |

## ログ・監査

| 要件ID | 要件 |
| --- | --- |
| NF-LOG-001 | ログイン失敗、権限エラー、状態不整合、想定外エラーをログに記録する |
| NF-LOG-002 | ログにはパスワード、パスワードハッシュ、セッションID、CSRFトークンを出力しない |
| NF-LOG-003 | 提出、承認、差戻し、再提出では、DB上の監査項目を更新する |
| NF-LOG-004 | 初回サンプルでは操作履歴テーブルは持たず、日報ヘッダに最新の監査項目を保持する |

## 性能・データ量

| 要件ID | 要件 |
| --- | --- |
| NF-PERF-001 | 初回サンプルの想定データ量は、社員20名、上長5名、案件20件、作業分類10件、日報12か月分を目安とする |
| NF-PERF-002 | 日報一覧、未承認一覧、月次集計、CSV出力では、検索条件に対象期間を指定できる |
| NF-PERF-003 | 一覧APIは初回サンプルではページングなしとするが、対象期間を絞って利用する前提とする |
| NF-PERF-004 | 月次集計は集計テーブルを持たず、承認済み日報と作業明細から都度集計する |
| NF-PERF-005 | CSV出力は対象年月単位とし、全期間一括出力は初回サンプルの対象外とする |
| NF-PERF-006 | 実装時はN+1、不要な全件取得、不要なDB更新を避ける |

## Oracle Database利用時の注意

| 要件ID | 要件 |
| --- | --- |
| NF-DB-001 | Oracle Databaseを前提に、ID採番、日時型、真偽値、予約語回避は `DB概念設計.md` に従う |
| NF-DB-002 | 文字列比較、日付範囲検索、年月検索でDB方言に依存する処理はRepository層に閉じ込める |
| NF-DB-003 | 日付検索では、業務日付と日時を混同しない |
| NF-DB-004 | 状態変更APIでは、状態更新と監査項目更新を同一トランザクションで処理する |
| NF-DB-005 | 同一社員・同一日の日報重複は、アプリケーション層とDB一意制約の両方で防ぐ |

## CSV出力

| 要件ID | 要件 |
| --- | --- |
| NF-CSV-001 | CSVはUTF-8 BOM付き、CRLF、カンマ区切りで出力する |
| NF-CSV-002 | 値にカンマ、改行、ダブルクォートを含む場合でもCSVとして壊れないように出力する |
| NF-CSV-003 | 月次集計CSVと日報明細CSVのヘッダー、ファイル名、値形式は `スコープ定義.md` に従う |
| NF-CSV-004 | 出力対象がない場合でもヘッダー行のみのCSVを出力できる |
| NF-CSV-005 | 承認済み日報のみCSV出力対象とする |

## 対象外

初回サンプルでは、以下は対象外とする。

- 多要素認証
- パスワード変更、パスワードリセット
- アカウントロック
- 操作履歴テーブル
- 通知、メール送信
- 添付ファイル
- 監査ログの長期保管
- バックアップ、リストア運用
- 冗長構成、災害対策
- 全期間の大量CSV一括出力

## AIレビュー重点観点

非機能レビュー用skillでは、少なくとも以下を確認する。

- 認可漏れがないか
- 入力値検証をフロントエンドだけに依存していないか
- 状態変更時に現在状態、ロール、対象データ権限を検証しているか
- SQLインジェクションのリスクがないか
- XSSにつながる表示実装がないか
- CSRF対策が不足していないか
- パスワードやセッション情報をログに出していないか
- N+1や不要な全件取得がないか
- 例外処理で内部情報を画面やAPIレスポンスに出していないか
- CSV出力で特殊文字を正しく扱っているか
