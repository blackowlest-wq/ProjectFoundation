# サンプルアプリ テスト設計書

## 目的

社内向け日報管理システムのサンプル開発におけるテスト方針、テスト範囲、確認観点、E2E相当確認シナリオを定義する。

本資料は、AIにテスト観点、テストケース、テストコード作成を依頼する際の入力情報として使用する。

## テスト方針

- 受入条件を満たしていることを、API確認、画面確認、E2E相当確認で確認する
- 主要機能はAPI確認と画面確認の両方を行う
- 提出、承認、差戻し、再提出、月次集計、CSV出力はE2E相当確認を必須とする
- 入力チェック、権限、状態遷移、休日区分、CSV形式を重点観点とする
- 画面確認では、`UIモック/index.html` と実装画面の構成・表示優先度・状態別操作が大きくずれていないことを確認する
- AI生成テストは人間が仕様根拠を確認してから採用する
- 欠陥を検出した場合は `../構想メモ/メトリクスと分類.md` の欠陥記録形式で記録する

## テスト種別

| テスト種別 | 目的 | 主な確認対象 | 実施方針 |
| --- | --- | --- | --- |
| 単体テスト | 個別ロジックの妥当性を確認する | 入力チェック、時間計算、集計ロジック、権限判定 | バックエンド中心に自動テスト化する |
| API確認 | API仕様と業務ルールの妥当性を確認する | リクエスト、レスポンス、HTTPステータス、状態更新 | 主要APIは正常系、異常系を確認する |
| 画面確認 | 利用者操作と表示制御の妥当性を確認する | 画面項目、入力制御、ボタン表示、エラー表示 | 主要画面はロール、状態、休日区分を変えて確認する |
| E2E相当確認 | 複数機能をまたぐ業務フローを確認する | 提出、承認、差戻し、再提出、集計、CSV出力 | 必須業務フローをシナリオとして確認する |

## 共通テスト観点

| 観点 | 確認内容 |
| --- | --- |
| 認証 | 未ログイン状態で保護された画面やAPIにアクセスできない |
| 権限 | 社員、上長、管理者の利用可能機能が仕様通りである |
| 入力チェック | 必須、形式、休日区分、休憩時間自動算出、勤務区分による時間内訳、作業明細合計のチェックが仕様通りである |
| 状態遷移 | `DRAFT`、`PENDING`、`REJECTED`、`APPROVED` の遷移が仕様通りである |
| エラー表示 | 入力エラー、権限エラー、状態不整合を画面で確認できる |
| APIステータス | 正常、入力エラー、権限なし、対象なし、状態不整合のHTTPステータスが仕様通りである |
| 集計 | 承認済み日報のみ集計対象になる |
| CSV | ヘッダー、文字コード、改行、区切り文字、ファイル名、特殊文字の扱いが仕様通りである |

## API確認観点

| No | 対象 | 正常系 | 異常系 |
| --- | --- | --- | --- |
| API-001 | ログイン | 正しいログインID、パスワードでログインできる | 誤った認証情報ではログインできない |
| API-002 | 日報登録 | 通常勤務の日報を登録できる | 必須不足、時間不整合、作業時間不一致では登録できない |
| API-003 | 日報編集 | `DRAFT`、`REJECTED` を編集できる | `PENDING`、`APPROVED` は編集できない |
| API-004 | 日報検索 | 権限範囲の日報を検索できる | 権限外の日報は検索結果に含まれない |
| API-005 | 日報詳細 | 権限範囲の日報詳細を取得できる | 権限外または存在しない日報は取得できない |
| API-006 | 提出 | `DRAFT` を `PENDING` に変更できる | `DRAFT` 以外、入力チェック違反、他社員日報は提出できない |
| API-007 | 承認 | 上長が承認対象グループの `PENDING` を `APPROVED` に変更できる | 承認対象グループ以外、`PENDING` 以外、社員、管理者は承認できない |
| API-008 | 差戻し | 上長が承認対象グループの `PENDING` を `REJECTED` に変更できる | コメント未入力、承認対象グループ以外、`PENDING` 以外は差戻しできない |
| API-009 | 再提出 | 社員が自分の `REJECTED` を `PENDING` に変更できる | `REJECTED` 以外、入力チェック違反、他社員日報は再提出できない |
| API-010 | 未承認一覧 | 上長が承認対象グループの `PENDING` のみ取得できる | 社員、管理者は利用できない |
| API-011 | 月次集計 | 管理者が承認済み日報を月次集計できる | 社員、上長は利用できない |
| API-012 | CSV出力 | 管理者が月次集計CSV、日報明細CSVを出力できる | 社員、上長は利用できない |

## 画面確認観点

| No | 対象画面 | 確認内容 |
| --- | --- | --- |
| UI-001 | ログイン画面 | ログイン成功、ログイン失敗、ログアウト後の保護画面アクセス不可 |
| UI-002 | 日報カレンダー・一覧画面 | 社員の当月カレンダー・一覧初期表示、社員向けグループプルダウン非表示、カレンダー状態色表示、カレンダーからの登録・編集・詳細遷移、ロール別表示範囲、日付・社員・状態・休日区分検索、権限外データ非表示 |
| UI-003 | 日報登録画面 | カレンダー入力、入力可能項目中心のヘッダ表示、休日区分ごとの入力可否、休憩区分・勤務区分表示、休憩時間自動算出、通常勤務・残業・深夜時間と作業時間合計の時分表示、保存、保存して提出 |

| UI-006 | 未承認一覧画面 | 上長の承認対象グループの `PENDING` のみ表示、承認・差戻しへの遷移 |
| UI-007 | 差戻し入力ダイアログ | コメント必須、差戻し確定、キャンセル |
| UI-008 | 月次集計画面 | 対象年月指定、4種類の集計表示、承認済みのみ反映 |
| UI-009 | CSV出力画面 | CSV種別選択、対象年月指定、ファイル出力 |

## 入力チェックテスト観点

| No | 観点 | 確認内容 |
| --- | --- | --- |
| VAL-001 | 通常勤務 | 勤務時刻、休憩区分、勤務区分、自動算出休憩時間、作業明細があり、実勤務時間と作業時間合計が一致する |
| VAL-002 | 休日作業なし | 勤務時刻、休憩時間、作業明細なしで登録できる |
| VAL-003 | 休日作業あり | 勤務時刻、休憩区分、勤務区分、自動算出休憩時間、作業明細があり、実勤務時間と作業時間合計が一致する |
| VAL-004 | 有給休暇 | 勤務時刻、休憩時間、作業明細を登録できない |
| VAL-005 | 午前休 | 勤務時刻、休憩区分、勤務区分、自動算出休憩時間、作業明細があり、実勤務時間と作業時間合計が一致する |
| VAL-006 | 午後休 | 勤務時刻、休憩区分、勤務区分、自動算出休憩時間、作業明細があり、実勤務時間と作業時間合計が一致する |
| VAL-007 | 時刻境界 | 勤務終了時刻が勤務開始時刻以前の場合は登録できない |
| VAL-008 | 休憩境界 | 休憩区分から算出した休憩時間が勤務開始から勤務終了までの時間以上の場合は登録できない |
| VAL-009 | 作業時間境界 | 作業明細の作業時間が0分以下の場合は登録できない |
| VAL-010 | 作業時間合計 | 作業明細合計が実勤務時間と一致しない場合は登録できない |
| VAL-011 | 勤務時間内訳 | 勤務区分により通常勤務、残業、深夜の時間が排他的に算出され、合計が実勤務時間と一致する |

## E2E相当確認シナリオ

| シナリオID | シナリオ | 主な確認内容 |
| --- | --- | --- |
| E2E-001 | 日報登録、提出、承認、月次集計 | 社員が通常勤務日報を登録・提出し、上長が承認し、管理者の月次集計に反映される |
| E2E-002 | 日報登録、提出、差戻し、修正、再提出 | 上長が差戻し、社員が差戻しコメントを確認して修正・再提出できる |
| E2E-003 | 承認済み日報のCSV出力 | 承認済み日報のみ月次集計CSV、日報明細CSVに出力される |
| E2E-004 | 権限外操作不可 | 社員が他社員日報を操作できず、上長が承認対象グループ以外の日報を承認・差戻しできない |
| E2E-005 | 状態不整合操作不可 | 古い画面や不正API呼び出しで、現在状態に合わない状態変更ができない |

## E2E-001 日報登録、提出、承認、月次集計

| 手順 | 操作主体 | 操作 | 期待結果 |
| --- | --- | --- | --- |
| 1 | 社員 | 通常勤務の日報を登録する | 日報状態が `DRAFT` になる |
| 2 | 社員 | 日報を提出する | 日報状態が `PENDING` になり、提出日時が記録される |
| 3 | 上長 | 未承認一覧を表示する | 提出された日報が表示される |
| 4 | 上長 | 日報を承認する | 日報状態が `APPROVED` になり、承認者と承認日時が記録される |
| 5 | 管理者 | 対象年月で月次集計を表示する | 承認済み日報の作業時間が集計される |

## E2E-002 日報登録、提出、差戻し、修正、再提出

| 手順 | 操作主体 | 操作 | 期待結果 |
| --- | --- | --- | --- |
| 1 | 社員 | 日報を登録して提出する | 日報状態が `PENDING` になる |
| 2 | 上長 | 差戻しコメントを入力して差戻す | 日報状態が `REJECTED` になり、差戻し情報が記録される |
| 3 | 社員 | 日報詳細または編集画面で差戻しコメントを確認する | 最新差戻しコメントを確認できる |
| 4 | 社員 | 日報を修正して保存する | 日報状態は `REJECTED` のまま保持される |
| 5 | 社員 | 日報を再提出する | 日報状態が `PENDING` になり、提出日時が更新される |
| 6 | 上長 | 再提出された日報を確認する | 未承認一覧に表示される |

## E2E-003 承認済み日報のCSV出力

| 手順 | 操作主体 | 操作 | 期待結果 |
| --- | --- | --- | --- |
| 1 | 社員、上長 | 承認済み日報を作成する | 対象年月に `APPROVED` の日報が存在する |
| 2 | 管理者 | 月次集計CSVを出力する | 指定ヘッダー、UTF-8 BOM付き、CRLF、指定ファイル名で出力される |
| 3 | 管理者 | 日報明細CSVを出力する | 日報明細単位で出力され、作業明細なしの休日・有給休暇も1日1行で出力される |
| 4 | 管理者 | カンマ、改行、引用符を含む備考の日報を出力する | CSVとして壊れずに出力される |

## 欠陥記録

テストで有効欠陥を検出した場合は、以下を最低限記録する。

- 欠陥ID
- 検出日
- 検出工程
- 対象機能
- 欠陥分類
- 重大度
- 再現手順
- 期待結果
- 実際の結果
- AI関与有無
- 対応状況

詳細な分類と記録形式は `../構想メモ/メトリクスと分類.md` を参照する。

## 完了条件

- 対象機能のAPI確認が完了している
- 対象機能の画面確認が完了している
- 提出、承認、差戻し、再提出、月次集計、CSV出力のE2E相当確認が完了している
- 主要な入力チェック、権限、状態遷移、休日区分、CSV形式が確認されている
- 検出した欠陥が欠陥記録に登録されている
