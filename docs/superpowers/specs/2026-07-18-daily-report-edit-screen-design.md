# 日報編集画面 設計書

## 目的

既存の日報登録・編集フォームを拡張し、設計書で定義された日報編集画面の状態別操作制御と差戻し情報表示を満たす。

## 対象範囲

### 実装対象

- `DRAFT`（未提出）の日報を編集・保存・提出できること。
- `REJECTED`（差戻し）の日報を編集・保存・再提出できること。
- `REJECTED` の日報で、最新差戻しコメント、差戻し者、差戻し日時を表示すること。
- `PENDING`（承認待ち）と `APPROVED`（承認済み）の日報では、入力、作業明細操作、保存、提出を無効化すること。
- 既存の更新・提出・再提出API、バックエンド認可、入力検証をそのまま利用すること。

### 対象外

- 日報詳細画面、承認画面、差戻し入力画面の新規実装。
- 新しいAPI、DBテーブル、DDL、承認状態の追加。
- React Routerなどのルーティングライブラリ導入。
- 既存の登録画面、日報一覧画面の機能変更。ただし編集画面と共有する型・表示部品の変更は対象とする。

## 現状と変更方針

日報詳細取得、更新、提出、再提出のAPIクライアントと、URLから日報IDを取得する既存の `DailyReportForm` は実装済みである。今回の変更は既存のフォーム構造を維持し、詳細レスポンスの差戻し情報を画面状態へ取り込み、承認状態に応じて操作可能範囲を制御する。

バックエンドは本人・状態・入力チェックを既に検証しているため、フロントエンドの無効化は利用者補助と誤操作防止を目的とする。認可と状態整合性の最終判定は引き続きバックエンドに置く。

## 画面設計

### 表示状態

| 状態 | フォーム項目 | 作業明細 | 保存・提出 | 追加表示 |
| --- | --- | --- | --- | --- |
| 新規 | 編集可 | 編集可 | 有効 | `日報登録`、`下書き` |
| `DRAFT` | 編集可 | 編集可 | 有効 | `日報編集`、`未提出` |
| `REJECTED` | 編集可 | 編集可 | 有効 | `日報編集`、`差戻しコメント`、`差戻し者`、`差戻し日時`、再提出 |
| `PENDING` | 読み取り専用 | 読み取り専用 | 無効 | `この状態の日報は編集できません。` |
| `APPROVED` | 読み取り専用 | 読み取り専用 | 無効 | `この状態の日報は編集できません。` |

編集URLの詳細取得中も入力操作を無効化し、既存値の取得前に新規フォームの初期値を保存してしまう競合を防ぐ。詳細取得に失敗した場合は既存のAPIエラー表示を使い、保存・提出操作は実行しない。

`DRAFT` の「保存して提出」は保存後に初回提出APIを呼び出し、`REJECTED` の「保存して再提出」は保存後に再提出APIだけを呼び出す。`REJECTED` から初回提出APIを呼び出さない。

### 差戻し情報

`REJECTED` のときだけ、次の値を表示する。

- 最新差戻しコメント: nullの場合は `-`
- 差戻し者: nullの場合は `-`
- 差戻し日時: nullの場合は `-`

値はA-009の詳細レスポンスに含まれる `rejectComment`、`rejectorName`、`rejectedAt` を使用し、クライアント側で履歴を推測・生成しない。再提出後の `PENDING` では編集画面の差戻し情報表示対象外であり、保持確認はA-009または日報詳細画面側の契約として扱う。

## データフロー

```text
編集URL
  -> DailyReportFormがreportIdを抽出
  -> fetchDailyReport(reportId)
  -> approvalStatusと差戻し情報をフォーム状態へ設定
  -> statusがDRAFT/REJECTEDなら編集操作を有効化
  -> statusがPENDING/APPROVEDなら全入力と保存・提出を無効化
  -> 保存時は既存のPUT API
  -> DRAFTの提出はsubmit API、REJECTEDの再提出はresubmit API
```

## エラー処理

- 詳細取得失敗: 既存の `日報の読み込みに失敗しました。` またはAPIの安全なメッセージを表示する。
- 保存・提出・再提出失敗: 既存のfield別エラー、APIメッセージ、汎用メッセージの優先順位を維持する。
- 画面表示後に別操作で状態が変わった場合: バックエンドの `409` を表示し、フロントの無効化だけで状態整合性を保証しない。状態競合を原子性まで保証する楽観ロック・条件付き更新の追加は今回対象外とし、別途バックエンド改善として保留する。
- CSRF、認証、所有者認可: 既存の共通APIクライアントとバックエンド認可へ委譲する。

## テスト設計

### フロント単体テスト

- `REJECTED` の詳細レスポンスから差戻しコメント、差戻し者、差戻し日時が表示される。
- `DRAFT` の保存・初回提出操作と、`REJECTED` の保存・再提出操作が有効である。`REJECTED` では初回提出APIを呼び出さない。
- `PENDING` と `APPROVED` の入力、明細追加・削除、保存・提出が無効である。
- 詳細取得中は編集操作が無効である。
- 差戻し情報がnullの場合に `-` が表示される。

### E2Eテスト

- 差戻し日報を編集画面で開き、差戻し情報を確認し、修正・再提出できる。
- 承認待ちまたは承認済み日報を編集URLで開いた場合、編集操作ができない。

### 回帰確認

- 既存の日報登録、下書き保存、初回提出、有給休暇入力、入力エラー、APIクライアント、バックエンド通常テストを実行する。
- TypeScript型チェック、Frontend build、Frontend lint、対象Markdownのlintを実行する。
