# コード・テスト・コミット・コメントの役割分担 設計書

## 1. 背景

既存の本体コードには、処理手順や分岐の動作を説明するコメントが多く、コード自体から読み取れる情報と、コードだけでは分からない設計判断が混在している。レビュー時に、コメントの有無だけでなく、どの情報をどの成果物へ残すべきかを判定できるルールが必要である。

## 2. 目的

次の役割分担を、作り込み・テスト・コミット・レビューの共通ルールにする。

| 成果物 | 残す情報 | 判定基準 |
| --- | --- | --- |
| 本体コード | How | 実装・構造・型・命名から処理方法が読めること |
| テストコード | What | テスト名、describe/it、Arrange/Act/Assertから確認対象と期待結果が読めること |
| コミットログ | Why | 背景、目的、利用者・保守への影響が読めること |
| コードコメント | How / Why not | 分岐・アルゴリズム・処理順・重要な不変条件・副作用などの誤読防止How、採用しなかった方式・制約・互換性・セキュリティ上の除外理由を残すこと |

## 3. 対象と対象外

### 対象

- 標準資料、AI向けプロンプト、ProjectFoundation用レビューskill
- `backend/src/main` と `frontend/src` の既存本体コードコメント
- `backend/src/test`、`frontend/test`、`frontend/e2e` のテスト表現
- 今後のコミットログと作業記録のテンプレート

### 対象外

- 既存コミット履歴の書き換え
- 生成済みの `frontend/playwright-report` などの成果物
- `oracle-test.example.properties` やCI設定など、実行手順そのものを伝える設定コメント
- 機能仕様、API、DB、画面の振る舞いの変更

## 4. 方針

### 4.1 本体コード

- ファイル名だけでは分からない責務、境界、公開契約を示す冒頭コメントは残す。
- 分岐の優先順位、アルゴリズム、処理順、重要な不変条件、副作用、外部API契約など、コードだけでは誤読しやすい箇所は`How:`で説明する。
- コードを逐語的に説明するだけのヘッダーコメントやインラインHowコメントは削除する。
- 採用しなかった方式や制約が判断の根拠になる箇所は、`Why not:`で対象方式と除外理由を書き換える。
- `Why not` は「何を採用しなかったか」と「なぜ採用しなかったか」を同じコメントで説明する。
- 仕様そのものをコメントへ転記せず、仕様書・テスト・命名・実装へ寄せる。
- ESLint等のツール指示コメントは、指示自体を維持し、直前に抑制理由を `Why not:` で残す。

例:

```ts
// How: 起動時の認証確認では401を通常の未ログイン状態としてnullへ変換し、その他の失敗は共通エラーとして伝播する。
// Why not: 未ログインを例外扱いするとログイン画面の初期表示をエラー扱いするため、nullで返す。
return null;
```

### 4.2 テストコード

- テスト名または `describe` / `it` の階層で、何を検証し、どの結果を期待するかを表す。
- 実装手順を説明するコメントは追加しない。
- テスト名だけで対象が分からない場合は、テスト名を変更する。コメントを追加する場合も、テストデータの選定理由などWhatを補う最小限にする。

### 4.3 コミットログ

- 件名は変更の要約、本文はWhyを記載する。
- 本文には、背景・目的・影響または採用判断を記載する。
- 実装手順や変更ファイルの列挙だけを本文にしない。
- 既存コミットは改変せず、今後のコミットと作業記録で運用を切り替える。

### 4.4 レビューと記録

- レビューでは、Howコメントの重複、Whatが読めないテスト、Whyのないコミット、根拠のないWhy notコメントを確認する。
- `Why not` を記載した場合は、対象方式、除外理由、再検討条件があれば作業記録にも残す。
- 既存コメントの移行で検出した問題は、作業記録と指摘一覧に記録する。

### テスト・記録

- 必要な指摘を `docs/AI活用開発研究/作業記録/日報登録編集_指摘一覧.md` へ追加

## 6. 受入条件

- 標準資料に4分類の定義、記載先、レビュー判定基準がある。
- 本体コードに、処理手順を逐語的に説明するだけのコメントが残っていない。必要な冒頭の責務・境界・契約、分岐・重要処理のHow、Why notの判断理由が残っている。
- テスト名またはテスト構造から、主要なテストの確認対象と期待結果が分かる。
- コミットログ運用に、Whyの記載項目と例がある。
- 作業記録に、コメント移行の判断、不採用方式、未実行確認を残す。
- 本体コード・テストコードの変更後に、バックエンド通常テスト、フロント単体テスト、型チェック、フロントビルド、E2Eを実行し、結果を記録する。
- 生成物や既存コミット履歴は変更しない。

## 7. 外部ドキュメント確認

今回の変更は、既存コードのコメントとプロジェクト内標準資料の更新であり、新しいライブラリ、フレームワーク、API、ツールの実装判断を含まない。そのため、外部ドキュメントの追加確認は不要と判断する。

## 8. 保留事項

- 既存コメントのうち、仕様の正式な根拠がコードと標準資料だけでは確認できないものは、Why notへ書き換えず、作業記録に保留として残す。
- 既存コミット履歴のWhy不足は、履歴を書き換えず、今後のコミットテンプレートとレビューで是正する。
