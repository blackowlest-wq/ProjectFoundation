# ログイン画面テスト補強 設計

## 目的

ログイン画面のAPI・部品単体テストだけでは確認できなかった、`App`の認証状態遷移とログインフォームのAPIエラー表示を自動テストで補強する。

## 対象

- `F-001-AC-002`: ロール別初期画面への遷移
- `F-001-AC-004`: 未認証時のログイン画面表示
- `F-001-AC-005`: ログアウト後の画面状態
- `F-001-AC-006`: API確認と画面確認
- `D-LOGIN-021`: App統合層のテスト不足

## 採用方式

既存のVitest + jsdom + React `createRoot`方式を使用する。

- `frontend/test/App.test.tsx`を新規作成する。
- `authApi`の通信関数だけをモックし、`App`、`LoginForm`、認証状態、URL更新は実コードを通す。
- 日報一覧・日報フォームは対象外のため、子コンポーネントをテスト内でモックする。
- `frontend/test/LoginForm.test.tsx`へ、APIエラーを`role=alert`へ表示するケースを追加する。
- Playwright E2E、Oracle接続、日報機能の検証は今回の対象外とする。

## 追加する確認

1. 未認証起動時にログイン画面が表示され、保護画面が表示されない。
2. 社員・上長・管理者のログイン成功時に、実際の`App`がロール別初期URLへ遷移し、利用者情報を表示する。
3. ログアウト成功時に利用者画面から`/login`へ戻る。
4. ログアウト失敗時に利用者画面を維持し、エラーを表示する。
5. ログインAPIがJSONエラーを返した場合、ログイン画面にエラーメッセージを表示する。

## 期待結果・副作用

- URLは`window.history.replaceState`後の`window.location.pathname`で確認する。
- 利用者画面ではユーザー名またはロール表示を確認する。
- ログアウト失敗では認証済み画面が残ることを確認する。
- API呼び出しは認証情報・CSRFの詳細を証跡へ出力しない。

## 完了条件

- 新規・追加テストが成功する。
- 既存Frontendテスト、lint、typecheck、buildが成功する。
- ケース台帳に新しい実テストIDと`D-LOGIN-021`の対応を記録する。
- `D-LOGIN-021`の状態を、追加テストの実行結果に合わせて更新する。
