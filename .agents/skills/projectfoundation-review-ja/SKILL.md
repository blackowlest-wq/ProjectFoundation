# ProjectFoundation Review

## 目的

実装後の確認を、汎用的な感想ではなく、受入条件・テストケース・実テスト・指摘・標準化判断まで追跡できる独立レビューとして完了する。

- `AGENTS.md`
- `docs/AI活用開発研究/構想メモ/標準化/標準化資料一覧.md`
- `docs/AI活用開発研究/構想メモ/標準化/開発フロー.md`
- `docs/AI活用開発研究/構想メモ/標準化/実装後レビュー表.md`
- `docs/AI活用開発研究/構想メモ/標準化/テスト・静的解析チェック表.md`

- 正本テストケース、受入条件レビュー、実装前レビュー結果、作業記録、指摘一覧

レビュー記録には、選択した優先度、適用理由、参照ファイル、参照節、担当、判定、根拠、記録先を残す。P1/P2を適用しない場合も、対象外理由または保留条件を記録する。

`作業記録テンプレート.md`と`メトリクスと分類.md`は、専門担当の個別レビュー入力ではなく、P0トレーサビリティ・統合担当が結果を集約するための資料として扱う。

## Hard Gate

観点別独立レビューと統合レビューが終わる前に、最終品質ゲートを完了扱いにしない。テストを追加・変更する場合は、Full実行前にテスト構成・責務分割の判定と記録を終える。Fullが成功していても、テストファイルの責務境界・テスト層・配置・共有supportの確認が未完了ならレビュー完了扱いにしない。

レビュー前にFullを実行した場合は「ベースライン確認」と記録し、指摘修正後にFull、E2E、coverage、Oracleなど対象範囲の最終確認を再実行する。

## 実装後の確認契約

- `実装後レビュー表.md`、`テスト・静的解析チェック表.md`、`テスト方針.md`、正本ケース、受入条件レビュー、実装前レビュー結果を入力とし、観点名だけの汎用レビューにしない。
- セキュリティ、原因分析、影響範囲など複数領域を確認する場合は、`superpowers:dispatching-parallel-agents` で独立に確認し、担当間で結論を共有せず最後に統合する。
- 外部レビューやサブエージェントの指摘は、`superpowers:receiving-code-review` を使って仕様・コード・テスト上の妥当性を確認してから対応する。鵜呑みにして修正しない。
- 主要機能完了後またはマージ前は、必要に応じて `superpowers:requesting-code-review` を使う。セキュリティ観点はCodex Securityの利用可否を判定する。
- 不具合や再現失敗は、`superpowers:systematic-debugging` により再現、原因候補、検証、修正、回帰テストの順で確認する。既存参照と影響範囲は検索で追跡する。
- 過去障害の全件照合は必須にせず、対象機能の回帰、同一原因、高リスク領域に関係する場合だけ参照する。

## 実装後レビューの確認内容

- 仕様、受入条件、対象外、入力検証、業務ルール、状態遷移、DB整合性、認証認可、CSRF、既存導線、ファイル配置、責務、共通化、テスト構成・責務分割、コメント、テスト、静的解析を確認する。
- 変更対象のテストを、テストファイル、責務、テスト層、対象本番モジュール、ケースID/実テストIDへ対応付ける。認証、一覧、登録・編集など独立責務の混在、Unit/UI/E2E/Oracleの不必要な混在、共有supportへの切り出し漏れを確認し、分割または維持の理由を記録する。
- 本体コードはHow、テストコードはWhat、コミット本文はWhy、コメントは必要なHow/Why notを担うことを確認する。`if`/`else`の処理手順は`How:`で示し、`Why not:`だけにしない。
- 指摘がない場合も、確認した観点と残るリスクを記録する。指摘がある場合は、仕様根拠、対応テスト、標準化判定、反映先または保留理由・再確認条件を記録する。

## 指摘対応契約

- 指摘ごとに妥当性、対応・保留・非対応、対応順、修正ファイル、確認テスト、標準化判定を整理する。
- 修正後は該当テストと必要な回帰テストを再実行し、最終品質ゲートを完了するまで完了扱いにしない。
- 標準化候補は、レビュー観点、チェック表、テストケース様式、受入条件、サンプル、テストデータのどこへ反映するかを具体化する。標準化しない場合は個別対応で十分な理由を残す。

## 手順

1. 差分、対象範囲、正本ケース資料、受入条件レビュー、実装前の保留事項を確認する。
2. ケースIDごとに、仕様根拠、期待結果、対象コード、実テストID/コマンドを照合する。正本ケースが更新されていなければ、レビュー完了扱いにしない。あわせてテスト構成・責務対応表を確認し、実テストが責務境界とテスト層に沿って配置されているかをFull実行前に判定する。
3. 上記の参照契約からレビュー範囲を選定し、各担当へ具体的な参照ファイルと節を明示して独立に実行する。P0は必須、P1/P2は適用条件に基づいて選定する。各担当には他担当の結論を渡さず、対応する専門プロンプトの出力形式を使う。「テスト不足」「セキュリティ」のような観点名だけを渡してはならない。
4. 2つ以上の独立レビューを依頼する場合は、**REQUIRED SUB-SKILL:** Use superpowers:dispatching-parallel-agents とし、最後に統合担当を置く。
5. 統合担当は重複、矛盾、重大度、仕様根拠、テスト要否、保留条件を整理し、1件ごとに判定する。
6. 指摘を修正し、修正したファイルと修正を確認するテストを記録する。保留は理由と再確認条件を必須とする。
7. テスト構成・責務分割の指摘を修正または保留判定した後に、対象テスト、回帰テスト、型チェック、lint、build、E2E、coverage、Oracleなどを計画どおり実行する。実行しない項目は理由と再確認条件を記録する。
8. 最終品質ゲートが成功した後に、作業記録、指摘一覧、レビュー記録、正本テストケース、受入条件レビューを揃える。

## 標準化判定

指摘を作業記録と指摘一覧へ転記するだけでは完了としない。各指摘について、今回限りか再発し得る一般観点かを判定する。

- 状態別操作、読込中/失敗時の安全側制御、UIと保存処理の判定共有は、画面機能の標準レビュー観点・テストケース様式へ反映する。
- null、境界値、異常系、APPROVEDなどのテスト不足は、テストケースレビュー観点と期待結果レビューへ反映する。
- 受入条件・ケース・実テストの紐付け不足は、トレーサビリティ様式へ反映する。
- 標準化しない場合は、個別事情である理由をレビュー記録へ残す。

受入条件ID、テストケースID、実テストID、指摘IDは別名前空間にし、同じ文字列を再利用しない。

## 完了条件

- 重大な仕様不整合、認証認可、CSRF、DB整合性、テスト不足が確認済みである。
- テスト構成・責務分割を確認し、分割・維持判断とその理由が記録されている。
- 必須の専門レビューが独立に実行され、統合結果が残っている。
- 正本ケース、受入条件、実テスト、指摘の対応が追跡できる。
- 指摘の標準化判定と反映先、または不採用理由が記録されている。
- 指摘修正後の最終品質ゲートが成功している、または未実行理由と再確認条件がある。
- 作業記録、指摘一覧、今回専用レビュー記録が更新されている。
- 本体コードはHow、テストコードはWhat、コミットログはWhy、コメントは必要なHow/Why notの役割分担になっている。
